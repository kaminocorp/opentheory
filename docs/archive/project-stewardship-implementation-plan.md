## 0. Global conventions & guardrails (apply to every task)

Keep these invariants in view — they're the ones easy to break:

- [x] **Export every new model** from `backend/app/models/__init__.py` `__all__` — Alembic's
  `env.py` discovers metadata via `from app.models import *`; a model missing from `__all__`
  is silently absent from autogenerated migrations.
- [x] **Do NOT register** `ProjectMember` / `ProjectInvitation` in `models/append_only.py` —
  they're mutable identity/governance rows (like `Account` and `Branch.status`), not ledger.
- [x] **Named PG enums:** add to `models/enums.py` as `StrEnum`, map with a stable
  `name=` (`project_role`, `invitation_status`). In **raw-SQL** migration predicates the enum
  **label is the uppercase StrEnum member name** (`'OWNER'`, `'PENDING'`) — lowercase errors.
- [x] **Declare table constraints on the model AND the migration** (esp. the partial unique
  owner index) so the test harness's `Base.metadata.create_all` builds exactly what Alembic
  installs (the `uq_actors_one_human_per_account` discipline).
- [x] **Authorization in the service layer:** route handlers stay thin; `ensure_can_manage`
  and invite/accept logic live in services. Unauth → `401`, signed-in non-member → `403`,
  missing → `404`.
- [x] **One transaction per write flow:** project-create writes Project + owner member +
  contribution and commits **once**; accept-invite writes the member + flips status in one
  commit.
- [x] **PII posture:** member/invitation public reads use `AccountSummary` (id + display_name +
  username) — **never** email/roles/external_id. `username` is a public handle and is OK to expose.
- [x] **Register routers** in `backend/app/api/router.py` next to the existing includes.
- [x] **Update `docs/changelog.md`** at the end of each release (per repo convention).
- [x] **Migrations are backend-owned**; chain `down_revision` correctly. *(⚠️ Actual chain:
  `0006_accounts → 0007_project_stewardship → 0008_account_username → 0009_account_email_index →
  0010_project_invitations` — the `0009` slot went to `0.8.4` hardening, so invitations became
  `0010`. Single linear head confirmed via `alembic heads`.)*

**Per-release verification gates (must pass before merge):**

```bash
# backend
cd backend && uv run ruff check . && uv run pytest        # DB tests need TEST_DATABASE_URL
# frontend
cd frontend && npm run typecheck && npm run lint && npm run build
```

> **DB tests:** per project policy we don't run destructive pytest against the live DB. Set
> `TEST_DATABASE_URL` to a **throwaway** Postgres only when explicitly greenlit; otherwise the
> DB-backed tests skip (suite still green) and we verify those flows against the live deploy.

## Release `0.8.2` — Account `@username` (prerequisite for invites)

> **⚠️ Shipped as `0.8.3`** (repo `0.8.2` was taken by stewardship hardening). Migration
> `0008_account_username` unchanged. Detail: `docs/completions/project-stewardship-0.8.3.md`.
> **File-location divergence:** the plan floated `core/usernames.py` for generation and
> `schemas/account.py`-or-`core` for the resolver; in the end `core/usernames.py` holds only the
> **pure, DB-free** helpers (`normalize` / `base_from` / candidates / `is_valid_username`), while
> the **DB-querying** `generate_unique_username` *and* `resolve_account_by_identifier` both live in
> `services/account.py` (the right layer for queries). Provisioning is split across
> `api/deps.py::_resolve_or_provision` (real first sign-in) and
> `services/account.py::create_account` (dev/test bootstrap).

**Goal:** every account has a unique, auto-generated, renameable public handle, exposed on
`/me` and `AccountSummary`, resolvable for invites.

### Backend

- [x] **T2.1 — Username generation util.** New `core/usernames.py`:
  `normalize(raw) -> str` (lowercase, strip to `[a-z0-9_]`, collapse, enforce 3–30, pad short),
  `base_from(email, display_name) -> str`, and a small reserved-set constant
  (`{"me","admin","system","accounts","api"}`). Pure functions, unit-tested DB-free.

- [x] **T2.2 — Model + provisioning.**
  - `models/account.py`: add `username: Mapped[str] = mapped_column(String(30), unique=True,
    nullable=False)`.
  - `api/deps.py::_resolve_or_provision`: when creating the Account, generate a candidate handle
    (`base_from(email, display_name)` → `normalize`), and on the unique-violation retry path,
    append a short suffix and retry (bounded). Generate **inside the existing first-login
    transaction**; the current `IntegrityError`→re-read handles the `external_id` race — extend it
    so a *username* collision regenerates rather than re-reads.
  - **Acceptance:** two provisions from emails with the same local-part get distinct handles.

- [x] **T2.3 — Schemas.**
  - `schemas/account.py`: add `username: str` to `AccountBase`/`AccountRead`; add `username` to
    `AccountSummary` (public handle — fine to expose). New `AccountUpdate(BaseModel)`:
    `username: str` validated to `^[a-z0-9_]{3,30}$`.
  - `schemas/account.py` or `core`: a `resolve_account_by_identifier(db, identifier)` helper —
    email-shaped (`"@" in identifier`) → match `lower(email)`; else → match `username`
    (lowercased). Returns the account or `None`. (Used by invites in `0.8.3`.)

- [x] **T2.4 — `PATCH /me`.** New route in `api/routes/me.py` (`ActingActor`): update the
  caller's **own** account `username`; `409` on unique collision (catch `IntegrityError`),
  `422` on pattern/reserved. Returns `MeRead`.

- [x] **T2.5 — Migration `0008_account_username`** (`down_revision="0007_project_stewardship"`,
  hand-authored backfill):
  1. `op.add_column("accounts", sa.Column("username", sa.String(30), nullable=True))`.
  2. **Backfill** every existing account with a unique handle. Recommended: do it imperatively
     in the migration via `conn = op.get_bind()` — read `(id, email, display_name)`, compute a

  3. `op.create_unique_constraint("uq_accounts_username", "accounts", ["username"])` and

  - **Ship with T2.2** — old provisioning code can't populate `username`, so the `NOT NULL` would
    break the next sign-in if code lags the migration.
  - `downgrade()`: drop constraint + column.

### Frontend

- [x] **T2.6 — Types + client.** Add `username` to `Account` / `Me` / `AccountSummary`

## Release `0.8.3` — Collaborators + invitation inbox (delivers C)

> **⚠️ Shipped as `0.8.7`** (repo `0.8.2`/`0.8.4`/`0.8.6` hardening + `0.8.5` jingle took the
> intervening slots). **Migration shipped as `0010_project_invitations`** (not `0009` — that slot
> went to `0.8.4`'s `0009_account_email_index`), chained onto it. Detail:
> `docs/completions/project-stewardship-0.8.7.md`. Post-merge follow-up `0.8.8` added `FOR UPDATE`
> row locks to `accept`/`decline`/`revoke` (invitee-side TOCTOU hardening) — see `docs/changelog.md`.

**Goal:** owner/admin invites an existing user by `@username` or email; the invitee sees it in a
top-right bell inbox and accepts (→ admin) or declines.

### Backend

- [x] **T3.1 — Enum + model.**
  - `models/enums.py`: `class InvitationStatus(StrEnum): PENDING; ACCEPTED; DECLINED; REVOKED`.
  - New `models/project_invitation.py` → `ProjectInvitation` (`IdMixin`, `TimestampMixin`),
    table `project_invitations`: `project_id` (FK `CASCADE`, indexed), `invitee_account_id` (FK
    `accounts.id` `CASCADE`, indexed), `role` (`project_role`, default `ADMIN`), `status`
    (`invitation_status`, default `PENDING`), `invited_by_account_id` (FK `SET NULL`, nullable),
    `UniqueConstraint("project_id","invitee_account_id", name="uq_project_invitation")`.
  - Export from `models/__init__.py`.

- [x] **T3.2 — Schemas.** `schemas/` new `invitation.py`: `InvitationCreate { identifier: str;
  role: ProjectRole = ADMIN }`; `InvitationRead { id, project_id, project_title, role, status,
  invited_by: AccountSummary | None, created_at }` (+ `invitee: AccountSummary` for the
  project-side list). PII-safe summaries only.

- [x] **T3.3 — Service.** New `services/invitations.py`:
  - `invite(db, project, identifier, role, actor)` — owner/admin (`ensure_can_manage`); resolve
    via `resolve_account_by_identifier` (`404` if none); reject self-invite + already-member
    (`409`); upsert the `(project, invitee)` row to `PENDING` (re-invite after decline resets);
    commit; return `InvitationRead`.
  - `my_pending(db, actor)` — caller's `PENDING` invitations (join project for title).
  - `accept(db, invitation_id, actor)` — invitee-only (`403` otherwise); one txn: create
    `ProjectMember(role=invitation.role, invited_by=invitation.invited_by_account_id)` (idempotent
    on `uq_project_member`), set status `ACCEPTED`; commit.
  - `decline(db, invitation_id, actor)` — invitee-only → `DECLINED`.
  - `revoke(db, project, invitation_id, actor)` — owner/admin → `REVOKED`.
  - `list_for_project(db, project, actor)` — owner/admin → pending invites.

- [x] **T3.4 — Routes.** New `api/routes/invitations.py` (+ register in `api/router.py`):
  - `POST /projects/{id}/invitations`, `GET /projects/{id}/invitations`,
    `DELETE /projects/{id}/invitations/{inv_id}` (all owner/admin).
  - `GET /me/invitations`, `POST /invitations/{inv_id}/accept`,
    `POST /invitations/{inv_id}/decline` (invitee).

- [x] **T3.5 — Migration `0009_project_invitations`** (`down_revision="0008_account_username"`,
  additive): create `invitation_status` enum + `project_invitations` table (FKs + unique). No
  backfill. `downgrade()` drops both.
  > **⚠️ Shipped as `0010_project_invitations`**, `down_revision="0009_account_email_index"`
  > (the `0009` slot was taken by `0.8.4` hardening). Also reuses the existing `project_role` enum
  > via `create_type=False`, and adds per-FK indexes. Otherwise as specified.

### Tests & done

- [x] **T3.9 — Tests** (`tests/test_invitations.py`): invite by username **and** by email →

## Appendix — sequencing & risk notes *(annotated with outcomes)*

- **Phase independence — held.** `0.8.1` shipped without usernames (member list showed
  `display_name`); `0.8.3` (`@username`) added handles; `0.8.7` consumed both
  (`resolve_account_by_identifier` + `ProjectMember`). The ordering was respected despite the
  renumbering.
- **Migration chain — final:** `0006 → 0007 → 0008 → 0009_account_email_index → 0010`. Each backend
  code change shipped **with** its migration (esp. `0008`'s `NOT NULL` username + the provisioning
  change). The `0009` slot was taken by `0.8.4` hardening, pushing invitations to `0010`.
- **Pre-existing nit — sidestepped (not fixed).** The frontend `ProjectStatus` type includes
  `"completed"`, which the backend enum lacks. Resolution: the `project-edit-form` status select
  offers **only the four backend statuses** (`draft`/`active`/`paused`/`archived`), so a `422` can't
  be triggered. A proper `completed`/lifecycle enum change remains a separate future task.
- **Email uniqueness — partially addressed.** `accounts.email` is still **not** uniqueness-enforced,
  so email-based invite resolution treats a rare multi-match as the most-recent account. `0.8.4`'s
  `0009_account_email_index` added a `lower(email)` index (findability/perf), but **not** a unique
  constraint — that remains an open follow-up if unambiguous email invites are wanted.
- **Live-DB safety:** backfills (`0007` none, `0008` username, `0010` none) are additive and
  idempotent. Deploys apply via `fly deploy` → `alembic upgrade head`; the DB-backed suites were
  **not** run locally (no-local-DB policy) and are the post-deploy live check.
- **TipTap bundle:** shipped lazy-loaded (`next/dynamic`, `ssr:false`); the read path uses only
  `react-markdown` + `remark-gfm` (raw HTML disabled), so public viewers don't pay for the editor.
