# 0.3.0 — Human-Operable Research Ledger

## Goal

Deliver the first end-to-end vertical slice of the OpenTheory research ledger so a human can manually perform a meaningful research action and see a durable, attributed checkpoint for it in the project UI.

This release does not introduce new domain primitives. It activates the primitives scaffolded in `0.1.0` (`Thread`, `Claim`, `Evidence`, `Checkpoint`, `Contribution`) by wiring write paths, services, and UI surfaces around them.

## Success Criterion

A user can:

1. Open or create a project.
2. Open a thread inside the project.
3. Add a structured claim to the thread.
4. Attach supporting, weakening, or contextual evidence to the claim.
5. Create an immutable checkpoint that records the resulting state change.
6. See the checkpoint in the project ledger UI with attribution and provenance.

Each meaningful action produces a `Contribution` record. Checkpoints are enforced append-only by a single service-layer chokepoint.

## Scope Summary

In scope:

- API routes and read models for `Thread`, `Claim`, `Evidence`, `Checkpoint`.
- A checkpoint service layer that enforces append-only ledger rules.
- Automatic `Contribution` records on meaningful user actions.
- The first real Alembic migration capturing the existing domain models.
- Frontend research workspace built on the project detail page: thread list, claim/evidence panel per thread, checkpoint timeline.
- Loading, empty, and error states for the new reads and writes.

Out of scope (deferred to later releases):

- Autonomous agents (`0.7.0`).
- Real payment processing (`0.6.0`+).
- Reputation or influence scoring.
- Validation records and branch lifecycle UI (`0.4.0`).
- Complex DAG visualization.
- Full authentication and authorization (`0.6.0`).
- Artifact storage for large uploaded files.
- Demo seed projects (`0.5.0`).

## Primitives Activated

| Primitive      | Already modeled | New in `0.3.X`                                                         |
| -------------- | --------------- | ----------------------------------------------------------------------- |
| `Project`      | yes             | enriched read model (threads, recent checkpoints)                       |
| `Thread`       | yes             | API routes, list/detail reads, create flow                              |
| `Claim`        | yes             | API routes, create flow scoped to a thread                              |
| `Evidence`     | yes             | API routes, create flow attaching to a claim with relation kind         |
| `Checkpoint`   | yes             | API routes, append-only service layer, ledger read model                |
| `Contribution` | yes             | auto-recorded for thread, claim, evidence, and checkpoint create flows  |

Not introduced in `0.3.X`: `Validation`, `Branch`, `Artifact` write flows, `FundingAllocation`, `Actor` identity beyond the existing model.

## Sub-Phases

The release is split into four sub-phases. Each ends in a deployable, demoable state.

### `0.3.1` — Backend Write Path: Threads, Claims, Evidence

Establish the create/read API surface for the three primitives that compose a research move, plus the two schema additions needed to make the relations queryable, without yet promoting any state change to a checkpoint.

Schema additions:

- Add `claim_evidence_links` join table:
  - `claim_id` (FK to `claims.id`, CASCADE)
  - `evidence_id` (FK to `evidence.id`, CASCADE)
  - `relation_kind` enum `evidence_relation_kind` (`support`, `weaken`, `context`)
  - `created_at` timestamp
  - composite PK on `(claim_id, evidence_id, relation_kind)` so the same pair can carry distinct relations if needed
  - indexes on `claim_id` and `evidence_id`
- Add `checkpoint_refs` join table (used by `0.3.2`, but introduced in the same migration to avoid a follow-up schema change):
  - `checkpoint_id` (FK to `checkpoints.id`, CASCADE)
  - `target_type` enum `checkpoint_ref_target` (`claim`, `evidence`, `artifact`, `thread`)
  - `target_id` UUID (no DB-level FK — polymorphic, validated in the service layer)
  - `role` `VARCHAR(40)` (e.g., `created`, `updated`, `referenced`)
  - composite PK on `(checkpoint_id, target_type, target_id, role)`
  - indexes on `(target_type, target_id)` for reverse lookups ("which checkpoints touched claim X?")

These two tables close the gaps identified in the current schema: there is no place today to record a claim ↔ evidence relation kind, and checkpoint references live only inside JSON columns and cannot be joined.

Backend:

- Add API routes:
  - `POST /api/v1/projects/{project_id}/threads`
  - `GET  /api/v1/projects/{project_id}/threads`
  - `GET  /api/v1/threads/{thread_id}`
  - `POST /api/v1/threads/{thread_id}/claims`
  - `GET  /api/v1/threads/{thread_id}/claims`
  - `GET  /api/v1/claims/{claim_id}`
  - `POST /api/v1/claims/{claim_id}/evidence` — creates an `Evidence` row and a `claim_evidence_links` row in one transaction with the supplied `relation_kind`
  - `GET  /api/v1/claims/{claim_id}/evidence` — joins through `claim_evidence_links` and returns evidence rows with their relation kind
- Add Pydantic schemas for create and read models, including the `relation_kind` field on evidence reads.
- Add a thin service layer for thread, claim, and evidence create flows.
- Generate the first real Alembic migration for the existing domain models **plus** `claim_evidence_links` and `checkpoint_refs`.
- Add focused tests:
  - thread creation under a project
  - claim creation under a thread
  - evidence attachment to a claim with each `support` / `weaken` / `context` relation
  - reading evidence for a claim returns the relation kind
  - read endpoints for each primitive

Frontend:

- No frontend changes in this sub-phase. Verified via API tests and manual `curl`/HTTP client.

Exit criteria:

- Migration applies cleanly against an empty database.
- All create and read endpoints return correctly shaped payloads, including the `relation_kind` on evidence reads.
- Tests for the four create flows pass.

### `0.3.2` — Checkpoint Service And Append-Only Enforcement

Make the ledger real. Introduce the checkpoint primitive as the only way to record a meaningful research state change, and enforce append-only behavior in one service-layer chokepoint.

Backend:

- Add `CheckpointService` with a single `create_checkpoint` entry point that:
  - validates the project and (optional) thread context
  - records the changed primitives by writing rows into `checkpoint_refs` (introduced in `0.3.1`), one per referenced claim, evidence, artifact, or thread, with a `role` describing the action
  - links parent checkpoints via `checkpoint_parents`
  - rejects any update or delete attempt against existing checkpoints
  - rejects any update or delete against `checkpoint_refs` rows (append-only by construction)
- Add API routes:
  - `POST /api/v1/projects/{project_id}/checkpoints`
  - `GET  /api/v1/projects/{project_id}/checkpoints`
  - `GET  /api/v1/checkpoints/{checkpoint_id}`
- Auto-record a `Contribution` for each successful checkpoint creation, referencing the actor, project, and target primitive.
- Auto-record a `Contribution` for thread, claim, and evidence create flows added in `0.3.1` (back-fill the contribution side now that checkpoints exist as the canonical event).
- Add focused tests:
  - checkpoint creation with parent linkage
  - checkpoint creation referencing a claim and evidence
  - rejection of update and delete attempts on checkpoints
  - contribution records exist for thread, claim, evidence, and checkpoint create actions

Frontend:

- No frontend changes in this sub-phase.

Exit criteria:

- A single service-layer entry point is the only producer of checkpoints.
- Append-only behavior is covered by tests.
- Contributions are written for all four create flows.

### `0.3.3` — Frontend Research Workspace

Surface the new write paths in the product. Expand the project detail page into a lightweight research workspace.

Frontend:

- Expand `/projects/[projectId]` into a workspace layout with three panels:
  - thread list (left): all threads in the project, with create-thread action
  - thread workspace (center): selected thread's claims and evidence, with create-claim and attach-evidence actions
  - checkpoint timeline (right): recent checkpoints for the project, newest first, with create-checkpoint action scoped to the current thread context
- Add typed API access for:
  - `GET /api/v1/projects/{project_id}/threads`
  - `GET /api/v1/threads/{thread_id}`
  - `GET /api/v1/threads/{thread_id}/claims`
  - `GET /api/v1/claims/{claim_id}/evidence`
  - `GET /api/v1/projects/{project_id}/checkpoints`
- Add typed write access for the four create flows.
- Add loading, empty, and error states for each panel.
- Keep create UIs minimal: a single-line title plus an optional description/notes field where appropriate. No rich editors.

Exit criteria:

- A user can complete the full success-criterion flow end-to-end through the UI without dropping to the API.
- Each panel handles loading, empty, and error cases.
- `npm run typecheck`, `npm run lint`, and `npm run build` pass.

### `0.3.4` — Ledger Read Model And Polish

Tighten the loop between actions and the visible ledger. The aim here is that the checkpoint timeline reads like a real research record, not a generic activity feed.

Backend:

- Extend the project detail read model to include:
  - threads (with claim counts)
  - the most recent checkpoints (bounded count)
  - aggregate counts (threads, claims, evidence, checkpoints)
- Extend the checkpoint read model to include:
  - the actor who created it
  - the contribution kind
  - referenced claim and evidence summaries (id, title, relation kind)
- Add focused tests for the enriched read models.

Frontend:

- Render the enriched checkpoint timeline: actor, action summary, referenced claims and evidence.
- Render project-level aggregate counts in the workspace header.
- Add empty-state copy that explains the next research move a user can make.
- Add a single end-to-end manual verification path documented in the changelog.

Exit criteria:

- The project page communicates the state of the ledger at a glance.
- A new user can understand what a checkpoint records without reading the docs.
- All previously listed tooling commands still pass on both backend and frontend.

## Cross-Cutting Concerns

- **Actor identity**: `0.3.X` does not add authentication. All actions are attributed to a single development actor seeded at startup or via a dev header. Real identity arrives in `0.6.0`.
- **Append-only invariants**: enforced exclusively in the checkpoint service. No route handler or migration is allowed to mutate existing checkpoints.
- **Schema migrations**: `0.3.1` produces the first real migration. Subsequent sub-phases add migrations only when they alter columns (e.g., relation kind enums, contribution target metadata).
- **Testing**: each sub-phase adds tests in the same commit that introduces the behavior. No sub-phase exits with a red suite.
- **Telemetry / logging**: out of scope. A single structured log line per checkpoint creation is sufficient for now.

## Definition Of Done For `0.3.0`

`0.3.0` ships when all four sub-phases are merged and the success criterion above is reproducible against a clean database from the UI alone.
