# Phase 0.3.4 — Ledger Read Model & Polish

**Status:** Complete. Backend `ruff` clean + `pytest` green (DB-backed tests skip without a DB); frontend `typecheck` / `lint` / `build` all pass.
**Release:** Fourth and final phase of `0.3.0 — Human-Operable Research Ledger`.
**Scope of this phase:** Backend read models + frontend rendering. Makes the checkpoint timeline read like a real research record and shows the state of the ledger at a glance. No new write paths, no new migration (read-model only — no column changes).

This document records what was built, where, and why, including the adversarial review pass.

---

## 1. Intent

Tighten the loop between the actions a user takes and what the project page shows back: aggregate counts at a glance, per-thread claim counts, and a checkpoint timeline enriched with *who* recorded it, *what action* it was, and *which* claims/evidence it touched. The plan source is `docs/executing/0.3.0-implementation-plan.md` (Phase 0.3.4).

---

## 2. What was built (file by file)

### Backend — enriched read models

- **`app/schemas/project.py`** — added `ProjectCounts` (threads/claims/evidence/checkpoints) and `ProjectOverview` (`ProjectRead` + `counts`).
- **`app/schemas/thread.py`** — added `ThreadSummary` (`ThreadRead` + `claim_count`) for the thread list.
- **`app/schemas/checkpoint.py`** — added `ActorSummary` (id, display_name, type); extended `CheckpointRefRead` with a resolved `label`; extended `CheckpointRead` with `author` (`ActorSummary | None`) and `contribution_kind` (`str | None`).
- **`app/services/projects.py`** (new) — `get_project_overview` returns the project plus four scalar aggregate counts (404 if the project is missing).
- **`app/services/threads.py`** — `list_threads` now returns `ThreadSummary` via a single grouped `OUTER JOIN` (`func.count(Claim.id)` grouped by thread), so threads with zero claims still appear with `claim_count = 0`.
- **`app/services/checkpoints.py`** — added `_enrich(db, checkpoints)`: batched (no N+1) resolution of authors (one `IN` query), `contribution_kind` (one query over `Contribution.checkpoint_id IN (...)`; unambiguous because only `create_checkpoint` sets `checkpoint_id`), and ref `label`s (one query per referenced target type, mapping `claim→statement`, `evidence→title`, `thread→title`, `artifact→name`). `create_checkpoint` / `list_checkpoints` / `get_checkpoint` all route through `_enrich`.
- **`app/api/routes/projects.py`** — added `GET /projects/{project_id}/overview` (distinct path template from `/{project_id}`, no shadowing).
- **`app/api/routes/threads.py`** — thread-list `response_model` is now `list[ThreadSummary]`.

### Frontend — rendering

- **`src/types/research.ts`** — added `ThreadSummary`, `ActorSummary`, `ProjectCounts`, `ProjectOverview`; extended `CheckpointRef` with `label` and `Checkpoint` with `author` + `contribution_kind`.
- **`src/lib/api.ts`** — added `getProjectOverview`; `listThreads` now returns `ThreadSummary[]`.
- **`src/lib/query-keys.ts`** — added the `overview(projectId)` key.
- **`src/components/workspace/project-workspace.tsx`** — header now shows the four aggregate counts (with a pulsing skeleton while the overview loads, `—` on error); fetches the overview; passes `projectId` into the claims panel for count invalidation.
- **`src/components/workspace/thread-list-panel.tsx`** — each thread shows a `claim_count` badge; creating a thread invalidates the overview.
- **`src/components/workspace/claim-list-panel.tsx`** — creating a claim invalidates the thread list (claim counts) and the overview; attaching evidence invalidates the overview.
- **`src/components/workspace/checkpoint-timeline-panel.tsx`** — each checkpoint renders the humanized `contribution_kind` action badge, the author ("by …"), the referenced primitives as `role → label` rows, stage, scope, and parent count; creating a checkpoint invalidates the overview; richer empty-state copy.

## 3. New / changed API surface

| Method & path | Response | Notes |
|---|---|---|
| `GET /api/v1/projects/{project_id}/overview` | `ProjectOverview` | project fields + `counts` (threads/claims/evidence/checkpoints); `404` if missing |
| `GET /api/v1/projects/{project_id}/threads` | `[ThreadSummary]` | now carries `claim_count` per thread |
| `GET /api/v1/projects/{project_id}/checkpoints` | `[CheckpointRead]` | each enriched with `author`, `contribution_kind`, and ref `label`s |
| `GET /api/v1/checkpoints/{checkpoint_id}` | `CheckpointRead` | same enrichment |

No request bodies changed; no new write paths; no migration (purely additive read fields).

## 4. Adversarial review pass

A multi-agent review (4 dimension reviewers → independent verification) ran over the backend+frontend diff: **10 raw findings, 9 "confirmed", 1 rejected.** Five of the nine "confirmed" were positive confirmations ("types correct", "invalidation correct", "null handling defensive") rather than defects. The genuinely actionable items were all fixed:

- **`contribution_kind` not rendered in the timeline (high).** The plan requires the timeline to show the *action summary*; the enriched field was sent by the backend and typed on the client but never displayed. **Fixed** — rendered as a humanized action badge next to the author.
- **Aggregate counts flashed `—` then jumped to numbers while the overview loaded (medium).** **Fixed** — a pulsing skeleton during load, `—` only on error.
- **Checkpoint empty-state copy didn't say *when* to record one (low).** **Fixed** — copy now gives concrete examples (propose a hypothesis, attach evidence, validate a result).

**Rejected (1)**, correctly: a claimed inconsistent result-set iteration in `_enrich` (the two iteration styles are both valid SQLAlchemy and correct for their result shapes).
