## 1. Intent

After `0.4.1`–`0.4.3` the write paths and basic surfaces exist, but the workspace can't yet answer "what's been assessed, what's contested, which lines are alive?" at a glance. This phase enriches the read side (no new writes, no schema change) and renders it, honoring plan **Decision #5**: a claim's display signal is *derived from validation history*, never a mutation of the stored `Claim.status`.

## 3. Design choices worth recording

- **Derived signal, not stored status (Decision #5).** `compute_signal` is pure and lives server-side; the contradiction set on the overview reuses it, so the per-claim chip and the project strip can never disagree. `Claim.status` is never written by a validation.
- **Explicit claim read construction.** Because `ClaimRead.validations` shadows the ORM `Claim.validations` relationship, the service must build reads field-by-field — `model_validate(orm_claim)` would async-lazy-load the relationship and break. This is the one non-obvious constraint introduced here.
- **One enrichment path for validations.** `validations_by_claim` wraps the existing `_enrich`, so recording-checkpoint/actor resolution is identical whether validations are read directly or embedded in a claim.
- **Embedded history replaces the separate fetch.** The frontend dropped the per-claim `/validations` query in favor of the embedded list — fewer requests, and the contested chip uses the authoritative server signal. The `GET /claims/{id}/validations` endpoint and its client binding remain for direct use.

## 5. Boundaries / what this phase intentionally does **not** do

- No new write paths and no migration (read-model only).
- No merge UI/logic (deferred, plan Decision #3).
- The contradiction rule is the simple "unretracted contradicts/failed" derivation; richer confidence machinery is a later release.
- No adversarial multi-agent review pass (can be run across the `0.4.x` diff if desired).
- `0.4.0` is now feature-complete across all four sub-phases; what remains before it fully "ships" per its Definition of Done is applying the migrations and running the DB-backed suites against a real database — still gated on the undecided DB target.
