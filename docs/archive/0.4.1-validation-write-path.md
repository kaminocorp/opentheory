# Phase 0.4.1 — Validation Write Path

**Status:** Code complete. DB-backed tests authored but **not yet run against a live database** (no Postgres chosen yet — Supabase vs. self-hosted still undecided, same posture as `0.3.1`–`0.3.4`).
**Release:** First of four phases in `0.4.0 — Validation And Branching`.
**Scope of this phase:** Backend only. Activates the `Validation` primitive (modeled since `0.1.0`, never written to) as a first-class, immutable assessment recorded **through** the existing checkpoint chokepoint. No branch write path (`0.4.2`), no frontend (`0.4.3`), no enriched project/claim read models (`0.4.4`).

This document records what was built, where, and why, and the exact steps to finish verification once a database exists. Plan source: `docs/plans/0.4.0-validation-and-branching.md` (sub-phase `0.4.1`).

---

## 1. Intent

Let a human **assess** what the ledger records: record a `Validation` against a claim, checkpoint, or branch with an outcome (`passed` / `failed` / `inconclusive` / `needs_reproduction` / `contradicts` / `retract`) and notes. Per `primitives.md` ("a `Validation` is recorded through `Checkpoint`") and the plan's **Resolved Decision #1**, a validation is a *ledger event*: recording one writes the `Validation` row **and**, in the same transaction, mints a checkpoint that references the validated target and the validation itself, attributed by a `validate` contribution.

The deliberate boundaries:
- The checkpoint service stays the **single producer** of checkpoints — the validation service composes with it, it does not write checkpoints directly (preserves the `0.3.2` chokepoint invariant).
- Validations are **immutable** — a re-assessment is a new row, never an edit (extends the append-only guard set).
- Validations do **not** mutate `Claim.status` (plan Decision #5 — confidence stays explainable from history; deferred to the read model in `0.4.4`).

## 2. What was built (file by file)

### Schemas (Pydantic v2) — new

- **`app/schemas/validation.py`** (new)
  - `ValidationCreate`: polymorphic target — `target_type` (`claim` / `checkpoint` / `branch` / `artifact`, validated in-service), `target_id`, `outcome` (`ValidationOutcome` enum), `notes?`. `project_id` comes from the path; `actor_id` from the header.
  - `ValidationRead`: `id`, `project_id`, `actor_id`, `actor` (`ActorSummary | None`, reused from `schemas/checkpoint.py`), derived `target_type` / `target_id`, `outcome`, `notes`, `recording_checkpoint_id`, timestamps.
  - Note: `recording_checkpoint_id` is the checkpoint the validation was *recorded through* (resolved from the `recorded`-role ref), deliberately distinct from the model's `Validation.checkpoint_id` column, which — when set — is instead the *target* checkpoint being validated. This ambiguity is the reason the recording link lives in `checkpoint_refs`, not on the row.

### Service layer — new

- **`app/services/validations.py`** (new) — the validation write/read path.
  - `_TARGET_MODELS` / `_TARGET_FK` map each `target_type` to its model (for existence/project checks) and to the typed FK column on `Validation` (`claim_id` / `checkpoint_id` / `branch_id` / `artifact_id`). `VALIDATION_TARGET_TYPES` is the public allow-set.
  - `create_validation`: validates project (404), target_type (422), target existence (404) and same-project (400); writes the `Validation` row with the matching typed FK set; `flush()`es for its id; then mints a checkpoint via `checkpoint_service.create_checkpoint(...)` with two service-supplied `extra_refs` — `{validated → target}` and `{recorded → validation}` — and `contribution_action="validate"`. The chokepoint owns the single commit, so the validation row, checkpoint, refs, and contribution persist atomically. When the target is a claim, the minted checkpoint is scoped to that claim's `thread_id` (optional-metadata nicety so it lands on the right line in the timeline); otherwise project-level.
  - `_enrich` resolves authors (one batched query) and the recording checkpoint per validation (one batched `checkpoint_refs` query on `target_type="validation"`, `role="recorded"`) — no N+1, mirroring the `0.3.4` checkpoint enrichment. `_derive_target` recovers `(target_type, target_id)` from whichever typed FK is set.
  - `list_validations` (project), `list_validations_for_claim` (claim), `get_validation` (404) — all go through `_enrich`.

### Service layer — edited (chokepoint extension)

- **`app/services/checkpoints.py`** — minimal, per plan Decision #1:
  - `create_checkpoint` gains two keyword-only params: `extra_refs: list[CheckpointRefInput] | None` (service-supplied, trusted, **not** re-validated — the calling service already validated the underlying rows in the same transaction) and `contribution_action: str | None` (overrides the recorded action; default still `create_checkpoint`). Refs written are now `[*payload.refs, *extra_refs]`; the contribution action is `contribution_action or ACTION_CREATE_CHECKPOINT`. The contribution's `target_type`/`target_id` remain the checkpoint (unchanged) — exactly the minimal surface the plan specified.
  - `_REF_TARGET_MODELS` / `_REF_LABEL_ATTR` extended with `branch` (label `name`) and `validation` (label `outcome`), so `checkpoint_refs` to these types validate and resolve human labels like any other primitive. This also widens `CHECKPOINT_TARGET_TYPES` (a client may now reference a branch/validation from a checkpoint).
  - Imports `Branch`, `Validation`, and `CheckpointRefInput`.

- **`app/services/contributions.py`** — added `ACTION_VALIDATE = "validate"`.

### Models — edited (append-only)

- **`app/models/append_only.py`** — added `Validation` to `_APPEND_ONLY_MODELS` (now `Checkpoint`, `CheckpointRef`, `FundingAllocation`, `Validation`). The same `before_update` / `before_delete` guards now refuse any ORM update/delete of a validation (plan Cross-Cutting "Append-only"). Docstring updated.

### API — new + edited

- **`app/api/routes/validations.py`** (new) — `POST /projects/{project_id}/validations` (requires `X-Dev-Actor-Id`), `GET /projects/{project_id}/validations`, `GET /claims/{claim_id}/validations`, `GET /validations/{validation_id}`. GET/POST only — no mutation methods (append-only at the API surface). Mounted at the root like threads/claims/evidence/checkpoints.
- **`app/api/router.py`** — include the validations router.

### Tests — new + edited

- **`tests/test_validations.py`** (new, DB-backed) —
  - `test_validate_claim_mints_checkpoint_with_refs_and_contribution`: one validation → exactly one checkpoint (claim-thread-scoped), both refs (`validated→claim`, `recorded→validation`), and one `validate` contribution linking that checkpoint.
  - `test_validate_checkpoint_target`: validating a checkpoint mints a *new* recording checkpoint (`recording_checkpoint_id != target`).
  - `test_validate_branch_target`: branch target exercised by inserting a `Branch` row directly (no branch write path until `0.4.2`).
  - `test_all_outcomes_accepted_and_listed`: every `ValidationOutcome` accepted; both list views return them; `recording_checkpoint_id` populated.
  - `test_validation_error_cases`: unknown outcome → 422, unknown target_type → 422, missing target → 404, cross-project target → 400, missing project → 404, missing dev-actor header → 400.
  - `test_validation_is_append_only`: ORM update/delete of a `Validation` raises `AppendOnlyError`; the row survives.
- **`tests/test_wiring.py`** (edited, DB-free) — added the validation create path to `WRITE_PATHS` (header-required) and the three validation read paths to the existence + no-mutation-methods assertions.

### No migration

`0.4.1` adds **no migration**. The `validations` table — with all four typed FK columns (`claim_id`, `artifact_id`, `checkpoint_id`, `branch_id`) and their indexes — is already in the `0001_baseline` migration (verified: `alembic/versions/0001_baseline.py` lines ~295–318). `checkpoint_refs` `target_type` is a free-string `VARCHAR`, so adding `validation`/`branch` to the vocabulary is a service-layer change only. (`checkpoints.branch_id` arrives in `0.4.2`.)

## 3. Design decisions realized (from the plan)

- **Decision #1 — recorded through a checkpoint.** Implemented exactly: `Validation` row + minted checkpoint with `validated`/`recorded` refs + `validate` contribution, all atomic via the chokepoint's single commit. The chokepoint extension is the minimal `extra_refs` + `contribution_action` surface the plan called for; the validation service never writes a `Checkpoint` itself.
- **Decision #4 — targets follow the model.** `claim` / `checkpoint` / `branch` / `artifact`; no `evidence` target (not in the model or `primitives.md`). `artifact` is wired through the identical code path but untested (no artifact write flow yet).
- **Decision #5 — no auto status mutation.** Nothing in this phase touches `Claim.status`; the derived "challenged" display signal is deferred to the `0.4.4` read model.
- **Append-only.** `Validation` joins the ORM guard set; the API exposes no PUT/PATCH/DELETE.

## 4. Verification

Run from `backend/`:

- `uv run ruff check .` → **clean.**
- `uv run pytest -q` → **5 passed, 24 skipped.** The DB-free wiring/health/read-model tests pass (including the new validation OpenAPI-surface assertions); the DB-backed tests (`test_validations.py` and the rest) **skip** because no `TEST_DATABASE_URL`/`DATABASE_URL` is configured — identical posture to `0.3.1`–`0.3.4`.
- Offline schema check: `Base.metadata` reports 14 tables; `validations` carries its full column set; `checkpoints` has **no** `branch_id` (confirming `0.4.1` is migration-free and `0.4.2` owns the schema change).

### To finish verification once a database exists

1. Set `DATABASE_URL` (or `TEST_DATABASE_URL`) to a reachable Postgres.
2. `uv run alembic upgrade head` (baseline `0001` + `0002` already cover every table this phase touches — no new revision).
3. `uv run pytest` — the 7 `test_validations.py` cases plus the existing DB-backed suites should pass green.

Manual smoke (with a project + thread + claim already created, and an actor id in `$A`):

```
POST /api/v1/projects/{project_id}/validations
  headers: X-Dev-Actor-Id: $A
  body: {"target_type":"claim","target_id":"<claim>","outcome":"passed","notes":"reproduced"}
→ 201; response has recording_checkpoint_id set.
GET  /api/v1/projects/{project_id}/checkpoints  → includes the minted checkpoint (action "validate").
GET  /api/v1/claims/{claim}/validations         → the validation, with actor + recording checkpoint.
```

## 5. Boundaries / what this phase intentionally does **not** do

- No branch write path, no `checkpoints.branch_id`, no branch-scoped checkpoints (`0.4.2`).
- No validation UI, outcome badges, or contradiction surfacing (`0.4.3`).
- No project/claim read-model enrichment (validation counts, validation history on claims, derived claim signal) (`0.4.4`).
- No `Claim.status` transitions from outcomes (Decision #5, deferred).
- No adversarial multi-agent review pass yet (prior phases ran one before merge; can be run on this diff if desired).
- Changelog `0.4.1` index entry not added here — left until the phase is confirmed, consistent with how `0.3.x` entries landed.
