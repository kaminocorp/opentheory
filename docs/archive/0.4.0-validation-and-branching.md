# 0.4.0 â€” Validation And Branching

## Goal

Add the next layer of research integrity on top of the `0.3.0` ledger: let a human **assess** what the ledger records (validation) and **explore a competing line without overwriting the main one** (branching). Both are already-modeled primitives (`Validation`, `Branch` shipped in `0.1.0`) that have no write path, service, routes, or UI yet. This release activates them through the same append-only checkpoint chokepoint established in `0.3.2`.

This release does not introduce new domain primitives. It activates `Validation` and `Branch`, adds one schema column (`checkpoints.branch_id`), and extends the checkpoint-ref vocabulary. It does **not** add merge resolution, funding, auth, or agents.

## Success Criterion

A user can:

1. Open a project with existing threads, claims, evidence, and checkpoints (the `0.3.0` flow).
2. Record a **validation** against a claim (or a checkpoint, or a branch) with an outcome (`passed` / `failed` / `inconclusive` / `needs_reproduction` / `contradicts` / `retract`) and notes.
3. See that validation produce a durable, attributed **checkpoint** in the ledger, and see the validation history surfaced on the validated primitive.
4. **Branch** from an existing checkpoint with a name and reason, opening a parallel line of exploration.
5. Record further checkpoints **on that branch** without touching the project's main line.
6. **Close** a branch as a dead-end (or superseded), with the closing reason preserved and visible â€” not deleted.
7. See contradictions (claims carrying a `contradicts` or `challenged` signal) and dead-end branches surfaced in the workspace.

Each meaningful action produces a `Contribution`. Validations and branch lifecycle events are recorded through the single checkpoint service. Validations are immutable once recorded.

## Scope Summary

In scope:

- `Validation` write path: a service that records a validation **through** the checkpoint chokepoint, API routes, read models, and a `validate` contribution.
- `Branch` write path: create-from-checkpoint, branch-scoped checkpoint creation, and close-as-dead-end/closed, all recorded through the checkpoint chokepoint.
- One migration: add `checkpoints.branch_id` (nullable; `NULL` = project main line).
- Extend the `checkpoint_refs` target vocabulary with `validation` and `branch`.
- Frontend: record-validation UI on claims/checkpoints, branch create/close UI, a branch selector that scopes the workspace, and read surfaces for validation history, contradictions, and dead-ends.
- Loading, empty, and error states for all new reads and writes.

Out of scope (deferred):

- **Merge resolution** (clean-merge / conflict / `merge-resolver`) â€” deferred; see Resolved Decision #3. `BranchStatus.MERGED` stays in the enum, unused.
- `Artifact` write flow â€” validation can *target* an artifact (the FK exists) but artifacts still can't be created, so artifact-validation is wired but only exercised once artifact writes land (`0.5.0`+).
- Evidence as a *validation target* â€” not in the `Validation` model or `primitives.md`; deferred (see Resolved Decision #4).
- Auto-transitioning `Claim.status` from validation outcomes â€” deferred (see Resolved Decision #5).
- Semantic `diff`/`blame` read operations over the DAG.
- Autonomous agents (`0.7.0`), payments (`0.6.0`+), reputation/influence scoring, full auth (`0.6.0`), demo seed projects (`0.5.0`).

## Resolved Decisions

The four questions raised when scoping this release, resolved with recommendations and rationale.

### Decision #1 â€” Validations are recorded *through* a checkpoint (first-class row + minted checkpoint)

`primitives.md` states a `Validation` "is recorded through `Checkpoint`" and "creates `Contribution`," and `research-flow.md` makes the validate stage a checkpoint-producing transition. So a validation is not merely a side row â€” it is a ledger event.

**Recommendation:** the validation create flow writes the `Validation` row **and**, in the same transaction, mints a `Checkpoint` via the existing `CheckpointService` chokepoint, then records one `validate` `Contribution`. The minted checkpoint carries `checkpoint_refs`:

- one ref to the validated target â€” `{target_type: "claim"|"checkpoint"|"branch"|"artifact", target_id, role: "validated"}`
- one ref to the validation itself â€” `{target_type: "validation", target_id, role: "recorded"}`

"Which checkpoint recorded validation X" is then answerable by a `checkpoint_refs` reverse lookup, and "which validations touched claim Y" by the `validated`-role refs â€” no new column on `Validation`.

This does **not** contradict `0.3.2`'s Resolved Decision #3 ("creates do not auto-promote to checkpoints"). That decision was about *authoring* content (thread/claim/evidence), which is not itself a belief-state change. A validation *is* an explicit assessment of the project's state of belief â€” the user's single explicit "validate" action mints the checkpoint, exactly as the explicit "record checkpoint" action does today. No background auto-promotion is introduced.

*Rejected alternative:* a bare `Validation` row + `validate` contribution, with the user separately choosing to checkpoint it. Rejected because it contradicts `primitives.md` ("recorded through Checkpoint") and lets validations exist outside the ledger.

*Implementation note (transaction composition):* the current `create_checkpoint` commits internally and always records a `create_checkpoint` contribution. Extend it minimally so callers can (a) pass extra service-supplied refs and (b) override the contribution action (default unchanged). The validation service adds the `Validation` row, flushes for its id, then calls `create_checkpoint(..., contribution_action="validate", extra_refs=[...])`; the chokepoint's single commit persists the validation row, the checkpoint, its refs, and the contribution atomically. The chokepoint stays the *sole* producer of checkpoints.

### Decision #2 â€” `NULL branch_id` is the main line (implicit default branch)

`research-git.md`: branches fork from a checkpoint and *contain later checkpoints*. Today `checkpoints` has no branch association, so there is no way to say which line a checkpoint is on.

**Recommendation:** add a nullable `branch_id` FK on `checkpoints` (â†’ `branches.id`, `ON DELETE SET NULL`). `branch_id IS NULL` means the checkpoint is on the project's **main line**. Branching from checkpoint `C` creates a `Branch` row (`forked_from_checkpoint_id = C`, `status = open`); checkpoints subsequently recorded on that branch carry its `branch_id`. Parentage continues to use `checkpoint_parents` (the first branch checkpoint's parent is `C`), so the DAG is unchanged â€” `branch_id` only *labels* which line a checkpoint belongs to.

*Rejected alternative:* an explicit default-`Branch` row seeded per project. Rejected because it forces every project create to mint a trunk branch, requires backfilling one for every existing `0.3.x` project, and buys nothing at this stage. "`NULL` = trunk" is a well-understood convention and is backward-compatible with all existing checkpoints (which become main-line automatically).

This is the only schema change in the release (migration `0003`).

### Decision #3 â€” Merge is deferred; branches *coexist* by default

`research-git.md` defines three branch outcomes: clean-merge, conflict (needs `merge-resolver`), and coexist. The roadmap lists branch *creation* and *dead-end/rejected* states but is silent on merge.

**Recommendation:** ship **create**, **coexist**, and **close** in `0.4.0`; defer **merge** (and conflict resolution) to a later release.

- **Coexist** is the default and needs no mechanic â€” sibling branches simply both exist and are both queryable.
- **Close** transitions a branch `open â†’ dead_end` (abandoned, with a recorded reason) or `open â†’ closed` (superseded), recorded through the checkpoint chokepoint with a `close_branch` contribution. Abandonment is recorded, never deleted (`primitives.md` invariant).
- **Merge** (unifying belief lines, semantic conflict resolution, the `merge-resolver` agent) is the most complex research-git operation and is not required to deliver validation+branching value. `BranchStatus.MERGED` remains reserved but unused.

This keeps `0.4.0` shippable and demoable while still honoring the "parallel exploration + preserved dead ends" invariant.

### Decision #4 â€” Validation targets follow the model: claim / checkpoint / branch (artifact wired, evidence deferred)

The `Validation` model's target FKs are `claim_id`, `artifact_id`, `checkpoint_id`, `branch_id` â€” matching `primitives.md` (which lists claim/artifact/checkpoint/branch). The roadmap's looser mention of "evidence" is not reflected in the model and is **not** added here (no `Validation.evidence_id`), avoiding a needless schema change; evidence is assessable indirectly via the claims it backs. Of the modeled targets, **claim**, **checkpoint**, and **branch** are exercisable in `0.4.0`; **artifact** validation is wired through the same code path but only covered by tests once an artifact write flow exists.

### Decision #5 â€” Validations do not auto-mutate `Claim.status`; status stays explainable from history

`primitives.md` requires confidence/status to be "explainable through evidence and validation history rather than a naked score." A validation with outcome `passed`/`failed`/`contradicts` therefore does **not** silently flip `Claim.status` in `0.4.0`. Instead the read model surfaces a claim's validation history and derives a *display* signal (e.g. "challenged" when an unretracted `contradicts`/`failed` validation exists). The stored `Claim.status` remains an explicitly-authored field. This avoids hidden mutation of a non-append-only row and defers confidence machinery to a later release. (`Validation` itself is treated as immutable â€” see Cross-Cutting Concerns.)

## Sub-Phases

Four sub-phases mirroring `0.3.0`: two backend, one frontend, one read-model + polish. Each ends deployable and demoable, and adds its tests in the same commit.

### `0.4.1` â€” Validation Write Path (Backend)

Activate `Validation` as a first-class, checkpoint-recorded assessment.

Backend:

- Extend `CheckpointService.create_checkpoint` minimally per Decision #1: accept service-supplied `extra_refs` and an optional `contribution_action` (default `create_checkpoint`). Keep it the sole checkpoint producer.
- Add `app/services/validations.py` with a single `create_validation` entry point that, in one transaction:
  - validates the project, the optional target (claim/checkpoint/branch/artifact must exist and belong to the project), and the outcome enum
  - writes the `Validation` row (`actor_id`, target FK, `outcome`, `notes`)
  - mints a checkpoint via the chokepoint, referencing the validated target (`role="validated"`) and the validation (`role="recorded"`), with `contribution_action="validate"`
- Extend the checkpoint-ref target vocabulary (`_REF_TARGET_MODELS` / `CHECKPOINT_TARGET_TYPES`) with `validation` and `branch` so the new refs validate and resolve labels.
- Add `app/schemas/validation.py` (create + read; read includes target type/id, outcome, notes, the recording checkpoint id, actor summary).
- Add API routes:
  - `POST /api/v1/projects/{project_id}/validations` (requires `X-Dev-Actor-Id`)
  - `GET  /api/v1/projects/{project_id}/validations`
  - `GET  /api/v1/claims/{claim_id}/validations`
  - `GET  /api/v1/validations/{validation_id}`
- Add `ACTION_VALIDATE = "validate"` to the contributions module.
- Add `Validation` to the append-only ORM guard set (see Cross-Cutting Concerns); expose **no** update/delete route.
- No migration (the `validations` table is in the `0001` baseline; ref target types are free-string `VARCHAR`).

Tests:

- validation against a claim mints a checkpoint with both refs and a `validate` contribution
- validation against a checkpoint and against a branch
- each `ValidationOutcome` value is accepted; an unknown outcome is rejected
- cross-project target rejected (400); missing target rejected (404)
- update/delete of a `Validation` raises `AppendOnlyError`; no mutation route exists
- the checkpoint chokepoint remains the only checkpoint producer (validation flow goes through it)

Exit criteria:

- Recording a validation produces exactly one checkpoint, two refs, one `validate` contribution, and one immutable `Validation` row.
- Append-only enforcement on validations is covered by tests.

### `0.4.2` â€” Branch Write Path (Backend)

Activate `Branch`: fork a line, record checkpoints on it, close it as a dead-end.

Schema (migration `0003_checkpoint_branch_id`, `down_revision = 0002_checkpoint_content`):

- `ADD COLUMN checkpoints.branch_id UUID NULL` with FK â†’ `branches.id` `ON DELETE SET NULL` and an index. `NULL` = main line (Decision #2). Safe/no-op on existing rows (all become main-line). Add the reverse `checkpoints` relationship on `Branch`.

Backend:

- Add `app/services/branches.py`:
  - `create_branch(project, from_checkpoint, name, reason, thread_id?)` â€” validates the fork checkpoint exists and is in-project; writes the `Branch` row (`forked_from_checkpoint_id`, `status=open`); mints a checkpoint through the chokepoint referencing the branch (`role="created"`) with `contribution_action="create_branch"`.
  - `close_branch(branch, outcome, reason)` â€” `outcome âˆˆ {dead_end, closed}`; sets `Branch.status`; mints a checkpoint referencing the branch (`role="closed"`) with `contribution_action="close_branch"`. Rejects closing an already-closed branch.
- Allow branch-scoped checkpoint creation: `CheckpointCreate` accepts an optional `branch_id`; the chokepoint validates it exists, is in-project, and is `open`, and stamps it on the checkpoint. `NULL` keeps main-line behavior unchanged.
- Add API routes:
  - `POST /api/v1/projects/{project_id}/branches` (fork; body carries `from_checkpoint_id`, `name`, `reason`)
  - `GET  /api/v1/projects/{project_id}/branches`
  - `GET  /api/v1/branches/{branch_id}` (includes its checkpoints, newest first)
  - `POST /api/v1/branches/{branch_id}/close` (body: `outcome`, `reason`)
- Add `ACTION_CREATE_BRANCH = "create_branch"` and `ACTION_CLOSE_BRANCH = "close_branch"`.
- `Branch` is mutable only for its `status` via `close_branch` (it is not append-only â€” it is a pointer/line object, like a git ref). Its lifecycle events are append-only because they are recorded as checkpoints.

Tests:

- branch forks from a checkpoint; the fork checkpoint and a `create_branch` contribution exist
- a checkpoint created with `branch_id` is stamped to the branch; one with `branch_id=NULL` stays main-line
- `close_branch` with `dead_end` sets status and records a `close_branch` checkpoint/contribution; closing an already-closed branch is rejected
- creating a checkpoint on a closed branch is rejected
- migration `0003` loads as head, renders valid Postgres DDL, and matches `metadata.create_all` (offline checks, consistent with `0.3.x`)

Exit criteria:

- A branch can be forked, extended with its own checkpoints, and closed as a dead-end, all through the chokepoint, with main-line checkpoints unaffected.
- Migration `0003` verified offline.

Exit criteria:

Make integrity legible: the workspace should show what's been assessed, what's contested, and what lines are alive vs. abandoned.

Frontend:

- Empty-state copy that explains the next integrity move (e.g. "No validations yet â€” assess a claim to record whether it holds").
- Document a single end-to-end manual verification path in `docs/changelog.md` (claim â†’ validation â†’ contradiction surfaced; checkpoint â†’ branch â†’ branch checkpoint â†’ close as dead-end).

Exit criteria:

- The workspace communicates validation state, contradictions, and branch/dead-end status at a glance.
- A new user understands what a validation and a branch record without reading the docs.
- All backend and frontend tooling commands still pass.

## Cross-Cutting Concerns

- **Single chokepoint preserved.** Validations and branch lifecycle events are recorded *through* `CheckpointService.create_checkpoint`; it remains the sole producer of checkpoints. The only extension is service-supplied refs + an optional contribution action.
- **Append-only.** `Checkpoint`, `CheckpointRef`, and `FundingAllocation` stay append-only. **Add `Validation`** to the ORM guard set: a validation is a point-in-time assessment feeding provenance, so a re-assessment is a *new* `Validation` row, never an edit. `Branch.status` is the one sanctioned mutable field (via `close_branch`), like a git ref moving; its transitions are themselves recorded as append-only checkpoints.
- **Actor identity.** Unchanged â€” all actions attributed via the `X-Dev-Actor-Id` dev header. Real identity arrives in `0.6.0`.
- **Schema migrations.** Only `0.4.2` migrates (`0003`, adding `checkpoints.branch_id`). `0.4.1` needs none (the `validations` table is in the baseline; ref target types are free-string). Verified offline, consistent with prior phases (no live DB chosen yet).
- **Confidence stays explainable.** No naked scores; no auto-mutation of claim status from validation outcomes (Decision #5).
- **Telemetry / logging.** Out of scope; a structured log line per checkpoint creation remains sufficient.

## Definition Of Done For `0.4.0`

`0.4.0` ships when all four sub-phases are merged and a user can, from the UI alone against a clean database: record a validation against a claim and see it as an attributed checkpoint with the claim's validation history; fork a branch from a checkpoint, record a checkpoint on it, and close it as a dead-end with its reasoning preserved and visible â€” with the project main line unaffected throughout.
