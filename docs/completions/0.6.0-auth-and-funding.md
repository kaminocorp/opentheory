**Status:** Code complete and **verified against a real Postgres**. The full backend suite runs green (`51 passed`, DB-backed tests *executing*, not skipping); both new migrations (`0004`, `0005`) apply, round-trip, and pass `alembic check` (no drift) against a live database; the frontend passes `typecheck` / `lint` / `build`; and the whole flow was exercised end-to-end over real HTTP against the migrated schema (see §7).

## 0. Prerequisite closed first — the `0.4.0` DoD gate

The plan names a prerequisite: the DB-backed tests must run **green against a separate test Postgres** before building auth/funding on the service layer ("building on it unverified is building on sand"). Through `0.4.7` that suite had only ever *skipped* (`5 passed, 31 skipped`), so the gate had never actually been exercised.

Closing it surfaced a **latent bug in the test harness itself**, exposed the first time the suite ran against a real database:

- **`tests/conftest.py` — schema teardown could not sort the FK cycle.** `branches` and `checkpoints` form a foreign-key cycle (`branches.forked_from_checkpoint_id → checkpoints.id` and `checkpoints.branch_id → branches.id`, the dual-FK from `0.4.2`). `Base.metadata.drop_all` cannot topologically order a cycle, so every test's teardown raised `CircularDependencyError`; leftover rows then cascaded into unique-violation failures in later tests (`27 failed, 31 errors`). Fixed by resetting the whole schema instead of dropping table-by-table: a `_reset_schema(conn)` helper runs `DROP SCHEMA public CASCADE; CREATE SCHEMA public` in setup (pristine start) and teardown. `create_all` already handles the cycle natively (it emits the cyclic FK as a post-create `ALTER`); only `drop_all` needed the workaround. This is **test-infrastructure only** — no app code — and it is what makes the `0.4.0` gate (and therefore `0.6.0`'s tests) actually runnable.

Result: `36 passed` on the pre-existing suite against a local throwaway Postgres, before any `0.6.0` code. Auth and funding are built on a *verified* service layer.

## 1. Intent

Give OpenTheory **real identity** then a **real funding write path**, in that order, without disturbing the service layer.

- **Identity** replaces the unverified `X-Dev-Actor-Id` header (any caller could act as any actor) with a verified Supabase JWT mapped onto the existing `Actor` model. The `ActingActor` *contract* (a resolved `Actor`) is preserved, so **no service signature changes**.
- **Funding** activates `FundingAllocation` — the last core primitive with a model but no write path — as a **money-stubbed, source-aware, contribution-only** ledger write. Native (Kamino-comped) funding is made real; Stripe and compute *spend* are deferred.

---

## 2. Decisions honored (from the plan)

- **#1 — identity maps onto `Actor.external_id`; contract preserved.** One `Actor` per human, `external_id = IdP subject`, `type = human`, JIT-provisioned on first login. Only the *internals* of `get_acting_actor` changed; no `User` table.
- **#2 — Supabase Auth behind a swappable adapter.** The backend only verifies a JWT and reads claims; all IdP coupling lives in `app/core/auth.py`. Swapping to Clerk/Auth0 changes only that file + config.
- **#3 — funding is contribution-only.** A funding event writes a `FundingAllocation` + one `fund` `Contribution` (`funding_allocation_id` set, `checkpoint_id = NULL`) — it **does not mint a research checkpoint**. The research DAG stays research-only. Verified live: `checkpoints == 0` after funding (§7).
- **#4 — `source` discriminator, native gated to `internal`.** New `FundingSource` (`native` | `stripe`) + `funding_allocations.source`. Native requires the `internal` role (`403` otherwise); the role is granted from an email allowlist on JIT provisioning.
- **#5 — append-only; allocations born in terminal status.** `native` → `settled`, `stripe` → `pending`. No in-place `pending → settled` mutation; `FundingAllocation` stays in the append-only guard set.
- **#6 — funding is authorization, not spend.** `funded = Σ settled`, `spent = 0`, `available = funded`. No debit ledger, no `OPENROUTER_API_KEY` here (both arrive with agents in `0.7.0`).

## 3. `0.6.1` — Verified Identity (backend)

### Schema / migration
- **`alembic/versions/0004_actor_roles.py`** (new, `down_revision = 0003_checkpoint_branch_id`) — `ADD COLUMN actors.roles ARRAY(String) NOT NULL DEFAULT '{}'`. The server default makes the backfill a no-op (existing actors get `[]`).
- **`app/models/actor.py`** — added `roles: Mapped[list[str]]` (`ARRAY(String)`, `server_default text("'{}'")`, `default=list`). A *mutable* identity attribute (like `Branch.status`), so it is **not** append-only guarded.

### Auth adapter + dependency
- **`app/core/auth.py`** (new) — the swappable seam. `verify_bearer_token(token) -> VerifiedIdentity(subject, email, display_name)` verifies an HS256 Supabase JWT (signature via the project JWT secret, plus audience + expiry + required `sub`/`exp`). Any failure raises `AuthError`. `supabase_jwks_url` is reserved as a forward hook for asymmetric keys.
- **`app/core/roles.py`** (new) — `INTERNAL_ROLE = "internal"` + `actor_is_internal(actor)`. Kept in `core` so the service layer can share it without importing upward from the API layer.
- **`app/api/deps.py`** (rewritten) — `get_acting_actor` now: extract `Authorization: Bearer` → if present, verify via `core/auth` (failure → `401`) → resolve `Actor` by `external_id == sub` → **JIT-provision** if absent (idempotent on the unique `external_id`, with an `IntegrityError` race recovery) → return. If **no** bearer token and `auth_dev_header_enabled`, fall back to the 0.3.x `X-Dev-Actor-Id` path (unchanged status codes: missing → 400, malformed → 400, unknown → 404). Otherwise `401`. Adds `require_internal` / `InternalActor` for route-level reuse. **Return type and every downstream service are unchanged.**

### Config / routes
- **`app/core/config.py`** — added `supabase_jwt_secret`, `supabase_jwks_url`, `supabase_project_url`, `supabase_jwt_audience` (default `"authenticated"`), `auth_dev_header_enabled` (default `False`), and `internal_actor_emails` (comma-split + lowercased via a validator, mirroring `backend_cors_origins`' `NoDecode` treatment).
- **`app/api/routes/me.py`** (new) — `GET /me` returns the resolved actor (incl. roles), driving the frontend identity menu; provisions on first authenticated call.
- **`app/api/routes/actors.py`** — `POST /actors` (the open bootstrap) now returns `404` unless `auth_dev_header_enabled` — retired in production, kept behind the dev flag for tests/local.
- **`app/schemas/actor.py`** — `ActorRead` gains `roles`; `ActorCreate` accepts optional `roles` (dev/test bootstrap can seed an `internal` actor).
- **`app/api/router.py`** — mounts the `me` router.
- **`pyproject.toml`** — added `pyjwt>=2.9.0`.

### Tests — `tests/test_auth.py` (new, DB-backed)
JIT provisions exactly one actor and is idempotent on `sub`; the `internal` email allowlist grants `["internal"]` (case-insensitive) vs `[]`; the `401` matrix (malformed / expired / wrong-audience / wrong-secret / missing-with-dev-off); an existing write attributes to the JIT actor; the dev-header path survives behind the flag and is `404` when off; the dev bootstrap can seed `internal`.

> `tests/conftest.py` sets `settings.auth_dev_header_enabled = True` process-wide so the existing `X-Dev-Actor-Id` suite keeps working; auth tests flip individual settings via `monkeypatch` (auto-restored).

## 4. `0.6.2` — Frontend Auth

- **`src/lib/supabase.ts`** (new) — null-safe Supabase browser client (`@supabase/ssr` `createBrowserClient`); `isSupabaseConfigured` is false when env is absent, so the app still builds/runs in dev mode.
- **`src/providers/auth-provider.tsx`** (new) — tracks the Supabase session, pushes its access token into the api client (`setAccessToken`), and exposes `signInWithEmail` (magic link), `signInWithGoogle` (OAuth), `signOut`.
- **`src/app/auth/callback/route.ts`** (new) — PKCE callback: exchanges the `code` for a session (server client + cookies) and redirects back.
- **`src/lib/api.ts`** (refactored) — the acting credential is now held **module-side** and synced by the providers, so the typed api functions stay plain. `request` attaches `Authorization: Bearer <token>` when set, else `X-Dev-Actor-Id` (dev mode); `writeInit(actorId, body)` → `writeInit(body)` and the `actorId` argument is dropped from **every** mutation. Added `getMe`, `listFunding`, `getProjectBudget`, `createFunding`.
- **`src/lib/use-identity.ts`** (new) — `useActingIdentity()` unifies the Supabase session + dev-actor into one credential, resolves the backend actor via `GET /me` (keyed by the active credential so switching identity refetches), and exposes `{ canWrite, isAuthed, isDev, isInternal, displayName, signInHint, hydrated }`. `AUTH_DEV` reads `NEXT_PUBLIC_AUTH_DEV`.
- **`src/providers/dev-actor-provider.tsx`** — now syncs the selected actor into the api client synchronously (`setDevActorId`) on hydration and on change (avoids a header race against TanStack refetching `/me`).
- **`src/components/shell/auth-menu.tsx`** (new) — signed-in identity (name + `internal` badge + sign-out) or a sign-in dropdown (Google + email magic link); renders nothing when Supabase is unconfigured (dev mode shows the switcher instead).
- **`src/components/shell/site-header.tsx`** — renders `<AuthMenu />`, and `<DevActorSwitcher />` only when `AUTH_DEV`.
- **`src/app/layout.tsx`** — wraps the tree in `AuthProvider` (inside `QueryProvider`, outside `DevActorProvider`).
- **The six write components** (`thread-list-panel`, `claim-list-panel` × 2 forms, `checkpoint-timeline-panel`, `branch-bar` × 2 forms, `validation-controls`) — swapped `useDevActor()` → `useActingIdentity()`, dropped `actorId` from every `mutationFn`/`mutate()`, gated `canSubmit` on `canWrite`, and replaced the "select an actor" hint with the auth-aware `signInHint`.
- **`src/types/research.ts`**, **`src/lib/query-keys.ts`** — added `roles` to `Actor`, a `Me` type, the funding types, `budget` on `ProjectOverview`, and `me` / `funding` / `budget` query keys.

## 5. `0.6.3` — Funding Write Path (backend)

### Schema / migration
- **`alembic/versions/0005_funding_source.py`** (new, `down_revision = 0004_actor_roles`) — creates the `funding_source` PG enum (`NATIVE` / `STRIPE` — the StrEnum member *names*, matching the baseline's enum convention) and adds `funding_allocations.source` NOT NULL. Adds the column with a temporary `server_default 'NATIVE'` to backfill any pre-existing rows, then **drops the default** so the column matches the ORM (no server default) and `alembic check` stays clean.
- **`app/models/enums.py`** — new `FundingSource(StrEnum)`.
- **`app/models/funding.py`** — added the `source` column (`Enum(FundingSource, name="funding_source")`).

### Service / schemas / routes
- **`app/services/funding.py`** (new) — `create_funding` validates the project, gates `source=native` on `actor_is_internal` (`403`), writes the allocation in its terminal status (`settled`/`pending`), records a `fund` contribution via `record_contribution(..., funding_allocation_id=…, checkpoint_id=None)`, and **owns the single commit** — no checkpoint minted. Plus `list_funding`, `get_funding`, and `project_budget` (one query, Python aggregation: `funded = Σ settled`, `spent = 0`, by-source/by-status breakdowns).
- **`app/schemas/funding.py`** (new) — `FundingCreate` (positive `Decimal` amount, 3-char currency upper-cased, defaults `kind=top_up` / `source=native`), `FundingRead` (incl. actor summary), `ProjectBudget`. (Monetary `Decimal`s serialize as JSON **strings**, e.g. `"500.00"`.)
- **`app/services/contributions.py`** — added `ACTION_FUND = "fund"` and a `funding_allocation_id` pass-through to `record_contribution` (Decision #3's note). The chokepoint is untouched.
- **`app/api/routes/funding.py`** (new) — `POST /projects/{id}/funding` (auth required; native gated in the service), `GET /projects/{id}/funding`, `GET /projects/{id}/budget`, `GET /funding/{id}`.
- **`app/services/projects.py`** + **`app/schemas/project.py`** — `get_project_overview` gains a `budget` block (reusing the single-query pattern). `funding_allocation` is deliberately **not** added to the checkpoint-ref vocabulary.

### Tests — `tests/test_funding.py` (new, DB-backed)
Native funding by an internal actor records one `settled` allocation + one `fund` contribution (`funding_allocation_id` set, `checkpoint_id` NULL) and **mints no checkpoint**; native by a non-internal actor → `403`; unauthenticated → `401`; stripe born `pending` and excluded from `funded`; append-only ORM enforcement on `FundingAllocation`; budget math + by-source/by-status; the funder/contributor/validator separation (only a `fund` contribution, no validation state touched); list + detail.

### How to reproduce the live path
1. Provision/confirm the DB; `cd backend && uv run alembic upgrade head` (through `0005`).
2. Run with `AUTH_DEV_HEADER_ENABLED=false`, `SUPABASE_JWT_SECRET` set, and `INTERNAL_ACTOR_EMAILS` listing your email.
3. Frontend: set `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY`, unset `NEXT_PUBLIC_AUTH_DEV`. Sign in → perform the `0.3.0`–`0.4.0` workflow as your verified self → native-fund a project (visible because your email is internal) → see budget + history update, a `fund` contribution exist, and **no checkpoint** minted; a non-internal user is `403` on native funding.

## 8. Cross-cutting & deferrals

- **Service-layer contract preserved** — auth changed only `get_acting_actor`'s internals; the single checkpoint chokepoint is untouched and funding stays out of it.
- **Append-only** guard set unchanged (`Checkpoint`, `CheckpointRef`, `FundingAllocation`, `Validation`); `actors.roles` is mutable identity, not a ledger event.
- **Deferred (unchanged from the plan):** real Stripe charges/Checkout/webhooks (`stripe` is modeled + creatable as `pending`, inert); compute metering / budget *debiting* + `OPENROUTER_API_KEY` (`0.7.0`); reputation/influence, branch merge, autonomous agents, per-funder API keys, object storage; the **`Artifact` write path** (still the other modeled-but-unwritten primitive).
- **Auth verification path** is HS256 against the Supabase JWT secret; asymmetric (JWKS) keys are a config + `core/auth.py` change away (`supabase_jwks_url` reserved).
