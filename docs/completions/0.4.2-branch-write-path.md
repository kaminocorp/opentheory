# Phase 0.4.2 — Branch Write Path

**Status:** Code complete. DB-backed tests authored but **not yet run against a live database** (no Postgres chosen yet — Supabase vs. self-hosted still undecided, same posture as `0.3.1`–`0.4.1`). Migration verified offline.
**Release:** Second of four phases in `0.4.0 — Validation And Branching`.
**Scope of this phase:** Backend + one migration. Activates the `Branch` primitive (modeled since `0.1.0`, never written to): fork a parallel line from a checkpoint, record checkpoints on it, and close it as a dead-end/superseded — all recorded through the existing checkpoint chokepoint. No frontend (`0.4.3`), no enriched project/claim read models (`0.4.4`), no merge (deferred, plan Decision #3).

Plan source: `docs/plans/0.4.0-validation-and-branching.md` (sub-phase `0.4.2`).

---

## 1. Intent

Let a human **explore a competing line without overwriting the main one**. A `Branch` forks from a checkpoint and contains later checkpoints; closing it preserves the reasoning rather than deleting it. Per the plan:

- **Decision #2 — `NULL branch_id` is the main line.** A checkpoint's `branch_id` is nullable: `NULL` = the project's main line; a value places it on a `Branch`. No seeded trunk row; all existing checkpoints become main-line for free.
- **Decision #3 — merge deferred.** Ship create + coexist (default) + close; `BranchStatus.MERGED` stays reserved/unused.
- Branch lifecycle events (fork, close) are recorded **through** the single checkpoint service, attributed by `create_branch` / `close_branch` contributions. The branch row itself is the one sanctioned mutable ledger object (its `status` moves, like a git ref); its *events* are append-only because they are checkpoints.

## 2. What was built (file by file)

### Schema / migration

- **`alembic/versions/0003_checkpoint_branch_id.py`** (new, `down_revision = 0002_checkpoint_content`)
  - `ADD COLUMN checkpoints.branch_id UUID NULL`, FK → `branches.id` `ON DELETE SET NULL`, plus index `ix_checkpoints_branch_id`. Nullable + no default ⇒ no-op backfill; existing checkpoints become main-line. `SET NULL` keeps the (append-only) checkpoint if a branch is ever removed.
  - Introduces a **second FK** between `checkpoints` and `branches` (the baseline already has `branches.forked_from_checkpoint_id → checkpoints.id`). Adding it via `op.add_column` with an inline FK is fine because both tables already exist — it emits a plain `ALTER TABLE … ADD CONSTRAINT`.

### Models — edited

- **`app/models/checkpoint.py`** — added the `branch_id` column (with the dual-FK note) and a `branch` relationship (`back_populates="checkpoints"`, `foreign_keys=[branch_id]`).
- **`app/models/branch.py`** — added the reverse `checkpoints` relationship (`foreign_keys="Checkpoint.branch_id"`) and **pinned `foreign_keys` on the existing `forked_from_checkpoint`** relationship (`"Branch.forked_from_checkpoint_id"`). This pinning is mandatory: once the second FK exists, SQLAlchemy can no longer infer the join for either relationship and the mapper fails to configure without explicit `foreign_keys`. (Verified with `configure_mappers()`.)

### Schemas (Pydantic v2)

- **`app/schemas/branch.py`** (new)
  - `BranchCreate`: `from_checkpoint_id`, `name`, `reason?`, `thread_id?`.
  - `BranchClose`: `outcome: Literal["dead_end", "closed"]` (so `merged`/anything else is a 422 at the schema boundary), `reason?`.
  - `BranchRead`: scalar fields incl. `status` (`BranchStatus`). `BranchDetail(BranchRead)` adds `checkpoints: list[CheckpointRead]` (the branch's checkpoints, newest first).
- **`app/schemas/checkpoint.py`** — added `branch_id: UUID | None` to both `CheckpointCreate` (default `None`) and `CheckpointRead`.

### Service layer

- **`app/services/branches.py`** (new)
  - `create_branch`: validates project (404), fork checkpoint (404 + same-project 400), optional thread (404/400); writes `Branch(status=open)`; `flush()`es; mints the branch's first checkpoint via the chokepoint — `branch_id=branch.id`, `parent_ids=[from_checkpoint_id]`, an `extra_ref` `{branch → created}`, and `contribution_action="create_branch"`. Atomic via the chokepoint's single commit.
  - `close_branch`: 404 if missing; **400 if not open** (already closed/dead-end/merged); flips `status` to `DEAD_END`/`CLOSED`; mints a close checkpoint referencing the branch `{branch → closed}` with `contribution_action="close_branch"`. **The close checkpoint is main-line** (`branch_id` omitted) by necessity — see §3.
  - `list_branches` (project, newest first), `get_branch` (detail incl. its checkpoints).
- **`app/services/checkpoints.py`** — `create_checkpoint` now validates and stamps `branch_id`: if provided, the branch must exist (404), be in-project (400), and be **open** (400 — a sealed line can't be extended). `_to_read` carries `branch_id`. Added `list_checkpoints_for_branch(db, branch_id)` (newest first, reusing `_enrich`) for the branch detail read. Imported `BranchStatus`.
- **`app/services/contributions.py`** — added `ACTION_CREATE_BRANCH` / `ACTION_CLOSE_BRANCH`.

### API

- **`app/api/routes/branches.py`** (new) — `POST /projects/{project_id}/branches` (fork), `GET /projects/{project_id}/branches`, `GET /branches/{branch_id}` (detail w/ checkpoints), `POST /branches/{branch_id}/close`. The two POSTs require `X-Dev-Actor-Id`.
- **`app/api/router.py`** — include the branches router.

### Tests

- **`tests/test_branches.py`** (new, DB-backed) — fork (first checkpoint stamped on the branch, parented on the fork point, `create_branch` contribution; fork point stays main-line; detail surfaces it); branch-scoped vs main-line checkpoints (`branch_id` set vs `None`); close as dead-end (status transitions, `close_branch` contribution, survives; already-closed → 400; checkpoint on a closed branch → 400); error cases (missing project/fork → 404, cross-project fork → 400, header required, `outcome:"merged"` → 422, close missing branch → 404).

## 3. Design / ordering decision worth recording

`create_checkpoint` (a) refuses a `branch_id` that is not open and (b) commits internally. Those two facts make it impossible to *both* stamp the close event onto the branch *and* keep the status-flip atomic: stamping requires the branch still open at mint time, but the mint commits before we could flip the status, splitting the operation across two transactions.

Resolution: the **close checkpoint is recorded on the main line** (`branch_id` omitted) and merely *references* the branch with role `closed`. The status flip (`open → dead_end/closed`) is left pending in the session and committed atomically by that same checkpoint write. Provenance is intact — "which checkpoint closed branch X" is a `checkpoint_refs` reverse lookup on `(target_type="branch", role="closed")`. The fork checkpoint, by contrast, *is* on the branch (the branch is open at that moment). This matches the plan ("mints a checkpoint referencing the branch") without weakening atomicity or the open-only-extend rule.

## 5. Boundaries / what this phase intentionally does **not** do

- No **merge** / conflict resolution; `BranchStatus.MERGED` stays reserved (plan Decision #3).
- No frontend: no branch selector, no dead-end markers, no validation/contradiction surfaces (`0.4.3`).
- No project/claim read-model enrichment (branch counts, validation history, derived signals) (`0.4.4`).
- The close checkpoint is main-line, not on the branch (§3) — deliberate, for atomicity.
- No parent linkage on the close checkpoint (kept minimal; the fork checkpoint does carry the fork point as parent).
- No adversarial multi-agent review pass yet (can be run on this diff if desired).
- Changelog `0.4.2` index entry not added here — left until the phase is confirmed, consistent with how `0.3.x`/`0.4.1` entries landed.
