# Phase 0.3.3 — Frontend Research Workspace

**Status:** Complete. `npm run typecheck`, `npm run lint`, and `npm run build` all pass.
**Release:** Third of four phases in `0.3.0 — Human-Operable Research Ledger`.
**Scope of this phase:** Frontend only. Expands `/projects/[projectId]` from a read-only detail page into a three-panel workspace where a user completes the full research move (thread → claim → evidence → checkpoint) through the UI, against the `0.3.1`/`0.3.2` API surface. No enriched read model (that is `0.3.4`).

This document records what was built, where, and why, including the adversarial review pass.

---

## 1. Intent

Surface the write paths in the product. A user should be able to open a project, decompose it into threads, record claims, attach evidence, and mark checkpoints — all without dropping to the API — with a dev-actor identity attached to every write. The plan source is `docs/executing/0.3.0-implementation-plan.md` (Phase 0.3.3).

---

## 2. What was built (file by file)

### Types & client

- **`src/types/research.ts`** (new) — TS mirrors of the backend read schemas: `Actor`, `Thread`, `Claim`, `Evidence` (with `relation_kind` + `link_id`), `Checkpoint` (with `parent_ids[]` + `refs[]`), plus the enum string unions (lowercase JSON values) and the create payload types.
- **`src/lib/api.ts`** (edited) — extended the single typed client with reads (`listThreads`, `getThread`, `listClaims`, `listEvidence`, `listCheckpoints`, `listActors`) and the four create flows (`createThread`, `createClaim`, `attachEvidence`, `createCheckpoint`) plus `createActor`. A `writeInit` helper attaches the `X-Dev-Actor-Id` header on writes; `createActor` deliberately omits it (bootstrap path). The shared `request` helper now surfaces the backend's `detail` field so mutation errors are legible.
- **`src/lib/query-keys.ts`** (new) — centralized TanStack Query keys so reads and the mutations that invalidate them never drift.

### Dev identity

- **`src/providers/dev-actor-provider.tsx`** (new) — a client context holding the selected actor id, persisted to `localStorage`. Reads storage only after mount and exposes a `hydrated` flag to avoid SSR/first-render mismatch.
- **`src/components/shell/dev-actor-switcher.tsx`** (new) — header dropdown that lists actors (`GET /actors`), lets the user pick one or create a new one (bootstrap, no header), and writes the selection through the provider. Shows `…` until hydrated.

### Workspace

- **`src/components/workspace/panel-state.tsx`** (new) — shared `PanelLoading` / `PanelError` / `PanelEmpty` so every panel renders the three states identically.
- **`src/components/workspace/thread-list-panel.tsx`** (new, left) — thread list + inline create-thread; selecting a thread drives the center/right panels.
- **`src/components/workspace/claim-list-panel.tsx`** (new, center) — claims for the selected thread + inline create-claim; each claim renders its own `ClaimEvidence` child that lists evidence (color-coded by relation kind) and has an inline attach-evidence form. Shows a clear empty state when no thread is selected.
- **`src/components/workspace/checkpoint-timeline-panel.tsx`** (new, right) — project checkpoint timeline (newest first) with a create-checkpoint form scoped to the current thread (sends `thread_id`), surfacing stage/refs/parents badges.
- **`src/components/workspace/project-workspace.tsx`** (new) — orchestrator: project header + the three-panel responsive grid (`lg:grid-cols-[280px_minmax(0,1fr)_340px]`), owns `selectedThreadId`.

### Wiring

- **`src/app/layout.tsx`** (edited) — wraps the app in `DevActorProvider` (inside `QueryProvider`).
- **`src/app/projects/[projectId]/page.tsx`** (edited) — renders `ProjectWorkspace` instead of the old read-only `ProjectDetail`.
- **`src/components/shell/site-header.tsx`** (edited) — replaced the placeholder account button with `DevActorSwitcher`.
- **`src/components/projects/project-detail.tsx`** (removed) — superseded by the workspace (the plan calls for expanding the detail page into the workspace).

## 3. Data flow & invalidation

| Read (query key) | Mutation | Invalidates |
|---|---|---|
| `["threads", projectId]` | `createThread` | `["threads", projectId]` |
| `["claims", threadId]` | `createClaim` | `["claims", threadId]` |
| `["evidence", claimId]` | `attachEvidence` | `["evidence", claimId]` |
| `["checkpoints", projectId]` | `createCheckpoint` | `["checkpoints", projectId]` |
| `["actors"]` | `createActor` | `["actors"]` |

All writes carry `X-Dev-Actor-Id` (the selected dev actor). Creating a thread auto-selects it; creating an actor auto-selects it.

## 4. Adversarial review pass

A multi-agent review (5 dimension reviewers → independent verification) ran over the diff: **8 raw findings, 3 confirmed, 5 rejected.** All three confirmed findings clustered on the actor/mutation lifecycle and were fixed across all four create forms:

- **Stale-`actorId` race (medium).** `mutationFn` captured `actorId` by reference, so switching actors while a write was in flight could send a stale/null header. **Fixed** — `actorId` is now captured by value and passed to `mutate(actorId!)`, evaluated synchronously at submit (where `canSubmit` already guarantees it is set).
- **Hydration flash of "Select an actor" (low).** The create forms showed the no-actor hint before `localStorage` hydration, so a returning user briefly saw a false error. **Fixed** — the hint is gated on `!actorId && hydrated`.
- **Double-submit on Enter while pending (low).** The submit handler checked `canSubmit` but not the pending state. **Fixed** — handlers now also require `!mutation.isPending`.

**Rejected (5)**, correctly: stale-selection / evidence-cleanup on *deletion* (no delete path exists in `0.3.x`), a claimed missing evidence loading/error state (it exists), and "evidence form too complex" (it is a compact single inline form — within the minimal-UI bar).

## 7. Hand-off to `0.3.4`

- The timeline currently shows `summary`, timestamp, stage, and ref/parent counts. `0.3.4` enriches the checkpoint read model (creating actor name, contribution kind, referenced claim/evidence summaries) — the timeline component is the render target for that.
- `0.3.4` also adds project-level aggregate counts (threads/claims/evidence/checkpoints) for the workspace header and empty-state copy explaining the next research move.
- The dev-actor switcher and the typed client are the stable host surface; `0.3.4` is additive rendering, not new write paths.
