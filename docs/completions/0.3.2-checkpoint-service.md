# Phase 0.3.2 — Checkpoint Service & Append-Only Enforcement

**Status:** Code complete. DB-backed tests authored but **not yet run against a live database** (no Postgres chosen yet — Supabase vs. self-hosted OVH still undecided, same posture as `0.3.1`).
**Release:** Second of four phases in `0.3.0 — Human-Operable Research Ledger`.
**Scope of this phase:** Backend only. Makes the ledger real: the checkpoint primitive becomes the only sanctioned way to record a meaningful state change, append-only is enforced at the ORM layer, and every create flow auto-records a contribution. No frontend (`0.3.3`), no enriched read models (`0.3.4`).

This document records what was built, where, and why, including the adversarial review pass and the exact steps to finish verification once a database exists.

---

## 1. Intent

Introduce `CheckpointService.create_checkpoint` as the single chokepoint for ledger writes, enforce the append-only invariant so corrections are *new* records (never edits/deletes), and back-fill automatic `Contribution` recording onto the `0.3.1` thread/claim/evidence create flows so every meaningful action is attributed. The plan source is `docs/executing/0.3.0-implementation-plan.md` (Phase 0.3.2).

The deliberate boundary: checkpoints are created **only** by explicit user action. Creating a thread/claim/evidence does **not** auto-promote to a checkpoint (plan Resolved Decision #3). Automated checkpointing by a review agent is a separate, later concern.

## 2. What was built (file by file)

### Models — edited

- **`app/models/checkpoint.py`**
  - Added `content` (`JSON`, NOT NULL, default `{}`): the free-form JSON payload a user authors on a checkpoint (plan Resolved Decision #2). The legacy research-git fields (`inputs`/`outputs`/`tool_invocations`/`evidence_refs`) remain untouched for later releases.
  - Made `stage` **nullable**: a research-flow `ThreadStage` is optional metadata, not platform law (`docs/primitives.md`), so a human may record a checkpoint without committing to a stage. The `0.1.0` model had it NOT NULL; this is the one substantive schema relaxation in the phase.
- **`app/models/__init__.py`** — exported `AppendOnlyError`; imports `append_only` (which registers the ORM guards on import).

### Models — new

- **`app/models/append_only.py`** (new)
  - `AppendOnlyError` and ORM `before_update` / `before_delete` mapper-event guards that raise on any attempt to update or delete an append-only row.
  - Guards cover `Checkpoint`, `CheckpointRef`, **and `FundingAllocation`** — CLAUDE.md / `docs/primitives.md` name both `Checkpoint` and `FundingAllocation` as append-only. `FundingAllocation` has no write path yet, so its guard is dormant but in place ahead of one landing (added during review — see §5).
  - `register_append_only_guards()` is idempotent (uses `event.contains`) and is called explicitly from `create_app()` so enforcement never depends on import order.

### Schemas (Pydantic v2) — new

- **`app/schemas/checkpoint.py`**
  - `CheckpointRefInput` (`target_type`, `target_id`, `role`) and `CheckpointRefRead`.
  - `CheckpointCreate`: `thread_id?`, `summary` (required, min length 1), `stage?` (`ThreadStage`), `content` (free-form dict), `notes?`, `parent_ids[]`, `refs[]`. `project_id` comes from the path; `author_id` from the header.
  - `CheckpointRead`: full read model including derived `parent_ids` and `refs`.

### Service layer — new

- **`app/services/contributions.py`** — `record_contribution(...)` adds a `Contribution` to the caller's session and **does not commit**, so the contribution shares the create's transaction (rolls back together). Action constants for the four flows live here.
- **`app/services/checkpoints.py`** — `create_checkpoint` is the **sole producer** of checkpoints:
  - Validates the project (404), optional thread (404 + same-project 400), parents (each must exist and share the project; duplicates deduped), and refs (each `target_type` ∈ `{claim, evidence, artifact, thread}` validated in-service; each target must exist and share the project).
  - Writes the `Checkpoint`, links parents via `checkpoint_parents`, writes one `checkpoint_refs` row per ref with its `role`, and records a `create_checkpoint` contribution — all in one transaction.
  - `list_checkpoints` / `get_checkpoint` eager-load `parents` + `refs` via `selectinload` and serialize through `_to_read`.

### Service layer — edited (contribution back-fill)

- **`app/services/threads.py`**, **`claims.py`**, **`evidence.py`** — each create now takes the acting `actor`, `flush()`es to assign the id, and records a contribution (`create_thread` / `create_claim` / `create_evidence`) **before** commit.

### API — new + edited

- **`app/api/routes/checkpoints.py`** (new) — `POST /projects/{project_id}/checkpoints` (requires `X-Dev-Actor-Id`), `GET /projects/{project_id}/checkpoints`, `GET /checkpoints/{checkpoint_id}`. Mounted at the root like threads/claims/evidence.
- **`app/api/routes/threads.py` / `claims.py` / `evidence.py`** — pass the resolved `actor` into the services.
- **`app/api/router.py`** — include the checkpoints router.
- **`app/main.py`** — call `register_append_only_guards()` in `create_app()`.

### Migration — new

- **`alembic/versions/0002_checkpoint_content.py`** (`down_revision = 0001_baseline`)
  - `ADD COLUMN checkpoints.content JSON NOT NULL` (safe: the baseline creates `checkpoints` empty, and the ORM supplies a `{}` default on insert — no backfill needed).
  - `ALTER COLUMN checkpoints.stage DROP NOT NULL`.
  - `downgrade` reverses both. References the existing `thread_stage` enum with `create_type=False` (no duplicate `CREATE TYPE`).

### Tests — new + edited

- **`tests/conftest.py`** — refactored into `db_engine` (creates/drops schema, skips without a DB) + `session_factory` + `client`, all sharing **one** engine so HTTP writes and direct DB assertions see the same data.
- **`tests/test_checkpoints.py`** (new, DB-backed) — checkpoint create with parents + refs; optional stage; header enforcement; full validation matrix (missing project/parent/ref, bad `target_type`, cross-project thread/ref); duplicate-parent dedup; **append-only ORM enforcement** (update + delete raise `AppendOnlyError`, with a selective negative test that a `Thread` updates freely); empty-refs semantics; contribution survival after a rejected delete; and contribution presence for **all four** create flows.
- **`tests/test_wiring.py`** (DB-free) — checkpoint paths exist, POST declares the dev-actor header, and the checkpoint paths expose **no mutation methods** (the API-surface half of append-only).

|---|---|---|---|---|

The three back-filled `0.3.1` write endpoints are unchanged in shape but now each record a `Contribution` in the same transaction.

## 4. Append-only — how it is actually enforced

Three layers, defense in depth:

1. **No mutation endpoints.** The checkpoint routes are GET/POST only. A `PUT`/`PATCH`/`DELETE` returns `405` from routing (asserted via the OpenAPI surface in `test_wiring.py`).
2. **ORM guards** (the real enforcement). `before_update` / `before_delete` mapper events on `Checkpoint`, `CheckpointRef`, and `FundingAllocation` raise `AppendOnlyError` the moment a tracked instance is flushed dirty or deleted — so the invariant holds even if a future service path or a bypassed route tries to mutate. Bulk Core `UPDATE`/`DELETE` and DDL (`drop_all` in tests) intentionally bypass the ORM unit-of-work; application writes all go through the ORM, which is the surface we protect.
3. **No service method mutates.** `CheckpointService` exposes only create + reads.

Consequence by design: because the guard blocks ORM deletes, an ORM-level cascade delete of a `Project` would be **refused** for its checkpoints (append-only wins over cascade). There is no project-delete path in `0.3.x`, so this is latent, intended, and documented.

## 5. Adversarial review pass

A multi-agent review (5 dimension reviewers → independent verification of each finding) ran over the diff: **19 raw findings, 6 confirmed, 13 rejected**. Actions taken:

- **`FundingAllocation` append-only gap (confirmed, high).** CLAUDE.md names it append-only, but the guard tuple omitted it. **Fixed** — added to `_APPEND_ONLY_MODELS` (zero behavior change today; no write path exists).
- **Guard registration depended on an `__init__.py` import side effect (confirmed, high — but premise partly wrong).** The verifier claimed guards were "never registered in production." That is **factually incorrect**: importing the submodule `app.models.checkpoint` executes the package `__init__.py`, which registers the guards (verified empirically by importing `app.main` alone). The underlying *fragility* was nonetheless real, so registration is now an **explicit, idempotent call in `create_app()`**.
- **Test coverage gaps (confirmed, medium/low).** Added tests: duplicate-parent dedup, empty-refs semantics, selective-guard negative test (`Thread` mutates freely), `AppendOnlyError` message assertion, and contribution survival after a rejected delete.
- **Rejected (13)** included several plausible-but-wrong concerns (post-commit `selectinload` "inconsistent state", transient many-to-many assignment, `MissingGreenlet` risk, contribution/checkpoint not rolling back together) — each refuted against the actual async-SQLAlchemy semantics. DAG cycle-checking and content-schema validation were correctly judged out of scope for `0.3.x`.

## 9. Hand-off to `0.3.3`

- The frontend consumes: `POST/GET /projects/{id}/checkpoints`, `GET /checkpoints/{id}`, plus the `0.3.1` thread/claim/evidence flows. All write requests carry `X-Dev-Actor-Id`.
- `CheckpointRead` exposes `parent_ids` and `refs` (with `role`) — enough for a timeline, though the enriched read model (actor name, contribution kind, referenced claim/evidence summaries) is `0.3.4`.
- Checkpoint creation is a deliberate user action scoped to the current thread context — surface it as an explicit "create checkpoint" affordance, not an automatic side effect of other writes.
