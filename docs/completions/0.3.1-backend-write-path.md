# Phase 0.3.1 — Backend Write Path: Threads, Claims, Evidence

**Status:** Code complete. Migration + DB-backed tests authored but **not yet run against a live database** (no Postgres chosen yet — Supabase vs. self-hosted OVH is undecided).
**Release:** First of four phases in `0.3.0 — Human-Operable Research Ledger`.
**Scope of this phase:** Backend + DB only. No checkpoint behavior, no contributions, no append-only enforcement (all `0.3.2`), no frontend (`0.3.3`).

This document records what was built, where, and why, plus the exact steps to finish verification once a database exists.

---

## 1. Intent

Stand up the create/read API surface for the three primitives that compose a single research move — **thread → claim → evidence** — plus the two relational tables needed to make those relations queryable, and the first real Alembic migration. The plan source is `docs/plans/0.3.0-implementation-plan.md` (Phase 0.3.1).

The deliberate boundary: this phase records *data*, not *ledger events*. A checkpoint is the canonical "meaningful state change" and is introduced in `0.3.2`; creating a thread/claim/evidence here does **not** produce a checkpoint or a contribution.

## 2. What was built (file by file)

### Models — new

- **`app/models/links.py`** (new)
  - `ClaimEvidenceLink` (`claim_evidence_links`): `IdMixin` + `TimestampMixin`, `claim_id` (FK → `claims`, `CASCADE`), `evidence_id` (FK → `evidence`, `CASCADE`), `relation_kind` `String(20)`. Unique constraint `uq_claim_evidence_relation` on `(claim_id, evidence_id, relation_kind)`. A **true many-to-many**: the same evidence row can back multiple claims, and a claim/evidence pair can be recorded under different relation kinds (per Resolved Decision #5 in the plan).
  - `CheckpointRef` (`checkpoint_refs`): `IdMixin` + `TimestampMixin`, `checkpoint_id` (FK → `checkpoints`, `CASCADE`), `target_type` `String(20)`, `target_id` `UUID` (no FK — polymorphic), `role` `String(40)`. **Introduced now but consumed in `0.3.2`**, to avoid a follow-up migration (plan Resolved Decision #8 / cross-cutting "Migrations").

### Models — edited

- **`app/models/claim.py`** — added `evidence_links` relationship (`back_populates="claim"`, `cascade="all, delete-orphan"`).
- **`app/models/evidence.py`** — added `claim_links` relationship (`back_populates="evidence"`, `cascade="all, delete-orphan"`).
- **`app/models/checkpoint.py`** — added `refs` relationship (`back_populates="checkpoint"`, `cascade="all, delete-orphan"`).
- **`app/models/__init__.py`** — imported and exported `ClaimEvidenceLink` and `CheckpointRef`. **Required**: `alembic/env.py` does `from app.models import *`, so a model missing from `__all__` is silently absent from migrations.

### Schemas (Pydantic v2) — new

- **`app/schemas/actor.py`** — `ActorBase` / `ActorCreate` / `ActorRead`.
- **`app/schemas/thread.py`** — `ThreadBase` / `ThreadCreate` / `ThreadRead`. `project_id` comes from the path, not the body.
- **`app/schemas/claim.py`** — `ClaimBase` / `ClaimCreate` / `ClaimRead`. `confidence` constrained to `[0, 1]`. `thread_id` and the derived `project_id` come from the path/parent.
- **`app/schemas/evidence.py`** — `EvidenceBase` / `EvidenceCreate` / `EvidenceRead`, plus a `RelationKind = Literal["support", "weaken", "context"]`. `EvidenceRead` includes the joined `relation_kind` and the `link_id`.

All read models use `ConfigDict(from_attributes=True)`.

### Service layer — new (`app/services/`)

- **`actors.py`** — `create_actor`, `list_actors`.
- **`threads.py`** — `create_thread` (404s if project missing), `list_threads`, `get_thread`.
- **`claims.py`** — `create_claim` (loads the thread, **derives `project_id` from it**, 404s if missing), `list_claims`, `get_claim`.
- **`evidence.py`** — `attach_evidence` (validates `relation_kind` against the authoritative `RELATION_KINDS` set, loads the claim, inherits its `project_id`/`thread_id`, creates `Evidence` + `ClaimEvidenceLink` in **one transaction** via `flush()` then `commit()`), `list_evidence` (joins `Evidence` ⨝ `ClaimEvidenceLink` and returns `EvidenceRead` rows carrying `relation_kind`).

Domain/validation logic lives here, keeping route handlers thin (per `CLAUDE.md` backend layering). Not-found and validation failures are raised as `HTTPException` from the service.

### API dependencies — new

- **`app/api/deps.py`**
  - `DbSession = Annotated[AsyncSession, Depends(get_db)]` (shared alias).
  - `get_acting_actor` / `ActingActor` — resolves the `X-Dev-Actor-Id` header to an existing `Actor`. Missing → `400`, malformed UUID → `400`, unknown id → `404`. There is **no authentication** in `0.3.x` (plan cross-cutting "Actor identity"; real identity arrives in `0.6.0`).

### Routes — new (`app/api/routes/`)

- **`actors.py`** — `POST /actors`, `GET /actors`. Mounted with `prefix="/actors"`. **Does not require an acting actor** (bootstrap path — you cannot have an actor to create the first actor).
- **`threads.py`** — `POST /projects/{project_id}/threads`, `GET /projects/{project_id}/threads`, `GET /threads/{thread_id}`.
- **`claims.py`** — `POST /threads/{thread_id}/claims`, `GET /threads/{thread_id}/claims`, `GET /claims/{claim_id}`.
- **`evidence.py`** — `POST /claims/{claim_id}/evidence`, `GET /claims/{claim_id}/evidence`.

Threads/claims/evidence span nested paths (e.g. `/projects/{id}/threads` **and** `/threads/{id}`), so their routers mount at the **root** (no prefix) and declare full paths themselves. All three write endpoints take `actor: ActingActor`, so the header is required and validated even though the actor is not yet stored anywhere (it becomes meaningful when contributions land in `0.3.2`).

### Router wiring — edited

- **`app/api/router.py`** — included the four new routers. `projects` and `actors` keep prefixes; `threads`/`claims`/`evidence` mount at the root.

### Migration — new

- **`alembic/versions/0001_baseline.py`** — single baseline revision (`down_revision = None`). Creates the 10 named enum types, then all 14 tables in FK-dependency order, then their indexes; `downgrade()` drops tables in reverse and then the enum types. See §4 for the design decisions that made this correct without a live DB.

### Tests — new (`tests/`)

- **`conftest.py`** — async `client` fixture (httpx `ASGITransport` + `AsyncClient`) bound to the app with a real test-DB session via a `get_db` dependency override. Creates all tables before each test and drops them after. **Skips** when no `TEST_DATABASE_URL`/`DATABASE_URL` is set or the DB is unreachable.
- **`test_wiring.py`** — DB-free OpenAPI checks: the new paths exist, write endpoints declare the `X-Dev-Actor-Id` header, and `POST /actors` does **not**.
- **`test_research_flow.py`** — DB-backed: actor create/list; the full thread→claim→evidence flow; every relation kind recorded and read back; invalid `relation_kind` → `422`; header enforcement (missing/malformed → `400`, unknown → `404`); create-under-missing-parent → `404`.

## 4. Key decisions & non-obvious findings

### 4.1 Enum labels are the StrEnum member **names** (UPPERCASE), not the values

The `0.1.0` models declare enums as `Enum(ProjectStatus, name="project_status")` **without** `values_callable`. By default SQLAlchemy persists each member's **name**, so the Postgres enum labels are `DRAFT`, `HUMAN`, `DECOMPOSE`, … — not the lowercase values (`draft`, …) that appear in the JSON API. (Verified offline by inspecting `column.type.enums`.) Round-tripping is consistent because SQLAlchemy maps DB `DRAFT` ↔ `ProjectStatus.DRAFT` ↔ JSON `"draft"`, so the **API stays lowercase while the DB stores uppercase**.

The baseline migration was written to match this exactly. **Recommendation (not done here):** before any data exists is the cheapest moment to decide whether the DB should store the lowercase values instead — that would mean adding `values_callable=lambda e: [m.value for m in e]` to every enum column and regenerating the baseline. Left as a deliberate, separate decision rather than a silent change. This affects the `0.1.0` models too (including the already-shipped `projects.status`).

### 4.2 The migration was made correct without a live database

A live Postgres was not available, so three offline checks substituted for `alembic upgrade head`:

1. **Script loads as head** — `alembic history` / `alembic heads` show `0001_baseline (head)` (catches syntax, imports, revision chain).
2. **DDL compiles** — `alembic upgrade head --sql` renders valid Postgres DDL with the expected `CREATE TYPE` labels.
3. **DDL matches the ORM** — the migration's emitted constraints/indexes were diffed against `metadata.create_all`'s output (compiled `CreateTable`/`CreateIndex` for the Postgres dialect). Two divergences were found and fixed so the baseline matches model metadata exactly (so future autogenerate runs come up empty):
   - `projects.slug` is `unique=True` **and** `index=True` → a single **unique index** (`ix_projects_slug`), **not** a separate unique constraint. The initial draft added a redundant `uq_projects_slug`; removed.
   - `actors.external_id` is column-level `unique=True` → an **unnamed** `UNIQUE (external_id)` (Postgres names it `actors_external_id_key`). The initial draft named it; made unnamed.
   - `uq_claim_evidence_relation` **is** named in the model's `__table_args__`, so the migration keeps that name.

Shared-enum handling: `thread_stage` is used by both `threads` and `checkpoints`. The migration creates each enum type **once** up front via `postgresql.ENUM(...).create(bind, checkfirst=True)`, and every column references the type with `create_type=False` so no duplicate `CREATE TYPE` is emitted.

### 4.3 Deferred enums stay as VARCHAR

`relation_kind`, `target_type`, and `role` are `VARCHAR` with service-layer validation (plan cross-cutting "Enums deferred"). `relation_kind` is additionally typed as a Pydantic `Literal` for an early `422`, but the **authoritative** allow-list is `RELATION_KINDS` in `app/services/evidence.py`.

### 4.4 Project ownership is server-derived

`claim.project_id` is taken from its thread and `evidence.project_id`/`thread_id` from its claim. Clients never supply `project_id`, so a claim can't be silently attached to the wrong project.

### 4.5 Test strategy: gated, not mocked

DB-backed tests run against a **real Postgres** (the models use `postgresql.UUID` and named enums, which SQLite can't represent), and skip when none is configured. This keeps the suite green today and makes it exercise the genuine schema the moment a `DATABASE_URL` is present — no SQLite-shaped divergence from production.

## 6. Remaining to fully close the phase (needs a database)

Once Supabase vs. self-hosted OVH Postgres is chosen:

1. `cp .env.example .env` and set `DATABASE_URL` to the async URL (`postgresql+asyncpg://…`).
2. `uv run alembic upgrade head` — apply the baseline (plan exit criterion: "applies cleanly against an empty database").
3. `export TEST_DATABASE_URL=…` (a scratch DB) and `uv run pytest` — the 6 DB-backed tests should now run and pass.
4. Smoke-test the flow with curl: create actor → project → thread → claim → attach evidence → list evidence (confirm `relation_kind`).

Until then, the phase's **code** is complete and offline-verified; only the live apply + DB-backed test run are outstanding.
