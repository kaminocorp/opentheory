# 0.5.0 — Demo Research Projects

## Goal

Turn the now-complete primitives into a *living research map*. The ledger write paths (`0.3.0`) and the integrity layer — validation and branching (`0.4.0`) — are all real, but every project a visitor opens is empty. `0.5.0` seeds the platform with two realistic, fully-developed research projects so the workspace reads like research in motion (active threads, supported and contested claims, pinned evidence, produced artifacts, recorded dead ends) instead of a scaffold waiting for input.

To make those demos credible, this release also closes the one remaining gap in the primitive set: **`Artifact` is the only core primitive with a model but no write path**. A minimal artifact write flow is added so demo projects can carry the plots, datasets, equations, and simulation outputs that physics and math research actually produce — and so the **artifact-validation path wired-but-untested since `0.4.1`** finally gets exercised.

This release does **not** introduce object storage for large files (Postgres still stores only hashes, metadata, and links — `CLAUDE.md`), real funding, auth, or agents. It activates `Artifact` writes, adds one join table + one migration, and delivers an idempotent, service-layer seed harness plus two authored demo datasets.

## Prerequisite (tracked separately, not a sub-phase of `0.5.0`)

`0.5.0` assumes a **live Postgres database with migrations applied**. Standing up the database, running `alembic upgrade head` against it, and getting the 31 DB-backed tests green is the still-open `0.4.0` Definition-of-Done gate (see every `0.4.x` changelog entry). It is a release/infrastructure task carried as the tail of `0.4.0`, **not** part of this plan's scope — but it is a hard blocker: you cannot seed demo projects into a database that has never been stood up. The first action before any `0.5.x` sub-phase ships is to confirm that gate is closed. (See Resolved Decision #1.)

## Success Criterion

A visitor who opens the deployed product, with no manual setup, can:

1. See **two seeded research projects** on the project index, each with a real research question and a populated overview (threads, claims, evidence, checkpoints, validations, branches, contradictions).
2. Open a project and read a coherent **research story** in the workspace: threads decomposing the question, claims at varying confidence, evidence pinned with citations, and a checkpoint timeline that shows who recorded what and why.
3. See the **integrity layer exercised on real content**: at least one **contested claim** (an unretracted `contradicts` validation), at least one **dead-end branch** with its reasoning preserved, and at least one **artifact carrying a validation**.
4. See **artifacts** as first-class produced objects (an exclusion plot, a dataset, a formal model) linked to the claims they express or support, content-addressed by hash.

The seed is reproducible from a single command against a clean (migrated) database, runs entirely **through the service layer** (so every seeded row carries the same provenance, append-only, and contribution guarantees as a human action), and is idempotent (safe to re-run).

## Scope Summary

In scope:

- **`Artifact` write path**: a service that creates an artifact **through** the checkpoint chokepoint, a claim↔artifact link, API routes, a read model, and a `create_artifact` contribution.
- One migration: add the `claim_artifact_links` join table (mirrors `claim_evidence_links`). This is the only schema change in the release.
- **`create_project` service**: extract project creation from the route into `services/projects.py` so the seed harness composes project creation uniformly with every other write (and records a `create_project` contribution — closing a small existing provenance gap).
- **Seed harness**: an idempotent, standalone script (`backend/scripts/seed_demo.py`, runnable as a module) that authors demo content by calling the existing services in sequence.
- **Two authored demo datasets**: dark matter model constraints and high-Tc superconductivity mechanisms, each exercising the full primitive set incl. a contradiction and a dead-end branch.
- Frontend polish only where real data exposes a gap (project index presentation, artifact rendering on claims, any empty-state copy that no longer applies once a project is full).

Out of scope (deferred):

- **Object storage for large files** — artifacts are metadata + URI + content hash only; binary upload/serving stays out (roadmap `0.5.0` explicitly excludes it; `CLAUDE.md` keeps blobs out of Postgres). Deferred to a later release if/when real uploads are needed.
- **Artifact *update* / re-version flow** — `0.5.0` ships artifact *create* only. Artifacts are modeled as updatable through a checkpoint (`primitives.md`), but the demo needs only creation; re-versioning is deferred.
- **Branch merge**, funding writes, reputation/influence, auth, autonomous agents — unchanged from `0.4.0`'s deferral list (`0.6.0`/`0.7.0`).
- **Generating real research** — the demo content is *authored* (curated to be realistic and internally consistent), not computed. Agents producing genuine findings is `0.7.0`.

## Primitives Activated

| Primitive      | Already modeled | New in `0.5.X`                                                                                  |
| -------------- | --------------- | ----------------------------------------------------------------------------------------------- |
| `Artifact`     | yes (`0.1.0`)   | the last write path: create-through-chokepoint service, claim link, API routes, read model      |
| `Claim`        | yes             | gains a `claim_artifact_links` association (artifact `expresses` / `supports` / `derived_from`)  |
| `Checkpoint`   | yes             | `create_artifact` flow mints a checkpoint referencing the artifact (`role="produced"`)           |
| `Validation`   | yes (`0.4.1`)   | the `artifact` target — wired in `0.4.1`, first *exercised* here against a seeded artifact        |
| `Contribution` | yes             | auto-recorded for `create_artifact` and `create_project`                                          |
| `Project`      | yes             | creation moves route → service and records a contribution                                        |

Not introduced in `0.5.X`: `FundingAllocation` write flow, branch *merge*, artifact *update*, object storage, `Actor` identity beyond the dev-header model.

## Resolved Decisions

### Decision #1 — The DB stand-up is a prerequisite, kept outside `0.5.0`'s scope

The roadmap names `0.5.0` "Demo Research Projects." Seeding is the first work that genuinely requires a live database, which is exactly the open `0.4.0` DoD gate. We keep these **separate**: standing up Postgres + applying migrations + greening the 31 DB-backed tests is tracked as the tail of `0.4.0` (a release/infra task), and `0.5.0` assumes it is done.

**Rationale:** the DB stand-up is infrastructure and validation of *already-written* `0.4.x` code, not a `0.5.0` feature; conflating them would let demo-content authoring hide a failing migration or a broken service. Closing the gate first means the seed harness is the *first thing that proves the whole stack works against a real database end-to-end* — a clean forcing function, but as a precondition, not a deliverable here.

*Consequence:* if greening the DB-backed suite surfaces real bugs in `0.3.x`/`0.4.x` services, those are fixed under the `0.4.x` line before `0.5.1` begins. `0.5.0` starts from a known-green DB.

### Decision #2 — Ship a minimal `Artifact` write path (metadata + URI + content hash), no object storage

`Artifact` is the only core primitive with a model (complete: `name`, `kind`, `uri`, `content_hash`, `artifact_metadata`, in the `0001` baseline) but no service, route, or schema. Validation already accepts an `artifact` target (`0.4.1`, Decision #4) but it has never been exercised because artifacts can't be created.

**Recommendation:** add a create-only artifact write path. An artifact is recorded with a name, a `kind` (free string, e.g. `plot` / `dataset` / `formal_model` / `simulation` / `proof`), an optional `uri` (where the bytes live — external for now), and a `content_hash` (the content-addressing anchor, `primitives.md`: "content-addressed where possible"). No bytes are uploaded or served by the backend.

*Rejected alternative:* defer artifacts entirely and seed only threads/claims/evidence/checkpoints/validations/branches. Rejected because (a) physics/math demos lean heavily on produced objects — an exclusion plot, a simulated phase diagram — and feel thin without them; (b) it would leave artifact-validation permanently untested; (c) it leaves the primitive set incomplete going into `0.6.0`. A *create-only, no-storage* path is the smallest version that closes the gap.

### Decision #3 — Artifacts are created *through* the checkpoint chokepoint, like every other ledger write

`primitives.md`: an artifact "may be created or updated through `Checkpoint`." So the artifact create flow composes with `CheckpointService.create_checkpoint` exactly as `create_validation` and the branch flows do (`0.4.x` pattern): the service writes the `Artifact` row, flushes for its id, writes the claim link(s), then calls the chokepoint with `extra_refs=[{target_type:"artifact", role:"produced"}, {target_type:"claim", role:"expressed"}]` and `contribution_action="create_artifact"`. One atomic transaction; the chokepoint stays the sole checkpoint producer and owns the single commit.

The checkpoint-ref vocabulary already includes `artifact` (label field `name`), so no ref-vocab change is needed. Add `ACTION_CREATE_ARTIFACT = "create_artifact"` to the contributions module.

**Append-only:** `Artifact` is **not** added to the ORM guard set. `primitives.md` models artifacts as updatable (re-versioned through a checkpoint), unlike `Checkpoint`/`CheckpointRef`/`FundingAllocation`/`Validation`. `0.5.0` ships create-only, but the model stays mutable for the deferred update flow — it is a content object, not a ledger event. (The *events* that touch it — the producing checkpoint, its contribution — are append-only as always.)

### Decision #4 — A `ClaimArtifactLink` join table (one migration), mirroring `ClaimEvidenceLink`

`primitives.md`: an artifact "may support or express one or more `Claim`." There is a `claim_evidence_links` table (with a `relation_kind`) but **no** claim↔artifact equivalent. To make the relationship first-class and queryable for the read model and UI, add `claim_artifact_links` mirroring the evidence pattern: `(claim_id, artifact_id, relation_kind)` with a uniqueness constraint, `relation_kind` a service-validated string (`expresses` / `supports` / `derived_from`).

This is **the only migration in `0.5.0`** (`0004_claim_artifact_links`, `down_revision = 0003_checkpoint_branch_id`). The artifacts table itself needs no migration (already in the `0001` baseline).

*Rejected alternative:* express the claim↔artifact association only via the producing checkpoint's refs (no join table). Rejected for fidelity and symmetry: evidence already has a dedicated relation-typed link, the read model and UI would otherwise have to reconstruct artifact↔claim ties from checkpoint refs (lossy, no relation kind), and the join table is a small, well-trodden pattern. Worth one migration.

### Decision #5 — Seeding is a standalone, idempotent, service-layer script — never a data migration, never Core inserts

The seed harness opens an `AsyncSession` and calls the existing services (`create_project`, `create_thread`, `create_claim`, `attach_evidence`, `create_artifact`, `create_checkpoint`, `create_validation`, `create_branch`, `close_branch`) in sequence, attributing actions to seeded demo actors.

**Why not a Core/SQL bulk insert:** it would bypass the append-only ORM guards *and* skip contribution recording — violating the `CLAUDE.md` rule that ledger writes go through the service layer. The whole point of the demo is that seeded provenance is *real*.

**Why not an Alembic data migration:** demo content is not schema. It must not auto-apply in every environment (tests, a fresh contributor's DB), it must go through services (migrations run raw SQL), and it should be explicitly invoked and re-runnable. So: a standalone script (`backend/scripts/seed_demo.py`), runnable as `uv run python -m scripts.seed_demo`.

**Idempotency:** the harness keys each demo project on a stable title; on re-run it detects an existing demo project and **skips** it (no duplicate). A `--reset` flag performs an explicit, dev-only teardown of demo data before reseeding — and because the append-only guards block ORM deletes by design, reset uses deliberate Core-level deletes scoped to the demo project ids (the same sanctioned-bypass posture as `drop_all` in tests, `CLAUDE.md`). Reset is guarded so it can only target seeded demo projects, never arbitrary data.

### Decision #6 — Two demo domains: dark matter constraints + high-Tc superconductivity

Both are roadmap-named candidates chosen because claims, evidence, and constraints are concrete and falsifiable (competing models, real observational/experimental constraints, genuine open contradictions). Each demo must exercise the **full** primitive set so the workspace shows the integrity layer at work, not just a happy path:

- multiple threads decomposing the question;
- claims spanning kinds (`hypothesis`, `constraint`, `observation`, `objection`, `result`) and statuses;
- evidence pinned per `primitives.md` (URI + retrieval timestamp + hash where available + citation metadata);
- at least one **artifact** (e.g. an exclusion-limit plot; a simulated phase diagram) linked to the claim it expresses, and **one validation targeting an artifact**;
- at least one **contested claim** (an unretracted `contradicts` validation) so the contradictions surface lights up;
- at least one **dead-end branch** with a preserved closing reason (a tried-and-abandoned line — e.g. a specific WIMP mass window now excluded);
- a checkpoint timeline that reads as a coherent narrative.

The *content* is the real deliverable of `0.5.3`–`0.5.4` and is authored for realism and internal consistency, not computed (Decision: out-of-scope "generating real research").

## Sub-Phases

Four sub-phases mirroring `0.4.0`: one backend (artifact write path), two for the seed harness + content, one frontend polish + live verification. Each ends deployable and demoable and adds its tests in the same commit. Unlike prior phases, `0.5.x` runs against the **live, migrated database** from the prerequisite — so each phase's tests are expected to actually execute (not skip).

### `0.5.1` — Artifact Write Path (Backend)

Close the last primitive gap so demos can carry produced objects and artifact-validation gets exercised.

Schema (migration `0004_claim_artifact_links`, `down_revision = 0003_checkpoint_branch_id`):

- Add `claim_artifact_links` (`claim_id`, `artifact_id`, `relation_kind`) with `UniqueConstraint(claim_id, artifact_id, relation_kind)` and indexes, mirroring `claim_evidence_links` (Decision #4). Add the reverse relationships on `Claim` and `Artifact`.

Backend:

- Add `app/services/artifacts.py` with `create_artifact(db, project, payload, actor, *, thread_id?, claim_links?)` that, in one transaction (Decision #3): validates project/thread/target-claim membership and `relation_kind`; writes the `Artifact` row; writes the `ClaimArtifactLink` row(s); mints a checkpoint via the chokepoint referencing the artifact (`role="produced"`) and any expressed claim(s) (`role="expressed"`), with `contribution_action="create_artifact"`.
- Add `app/schemas/artifact.py` (`ArtifactCreate`, `ArtifactRead`; read includes kind, uri, content_hash, linked claims with relation kind, the producing checkpoint id, actor summary).
- Add API routes: `POST /api/v1/projects/{project_id}/artifacts` (requires `X-Dev-Actor-Id`), `GET /api/v1/projects/{project_id}/artifacts`, `GET /api/v1/artifacts/{artifact_id}`, `GET /api/v1/claims/{claim_id}/artifacts`.
- Add `ACTION_CREATE_ARTIFACT = "create_artifact"` to the contributions module.
- Export the new model/relationships from `models/__init__.py` (Alembic discovery — `CLAUDE.md`).

Tests (now actually running against the live DB):

- creating an artifact mints exactly one checkpoint (refs: artifact `produced` + claim `expressed`), one `create_artifact` contribution, one `Artifact` row, and the `ClaimArtifactLink`(s);
- cross-project claim link rejected (400); unknown `relation_kind` rejected; missing target (404);
- a **validation against the created artifact** succeeds end-to-end (closes the `0.4.1` artifact-target test gap);
- the checkpoint chokepoint remains the only checkpoint producer;
- migration `0004` loads as head and round-trips (offline + against the live DB).

Exit criteria:

- An artifact can be created through the chokepoint, linked to a claim, and validated — all attributed and append-only-safe — and the `0.4.1` artifact-validation path is covered by a passing test.

### `0.5.2` — Seed Harness + Project Service (with the first demo)

Build the reproducible seeding machinery and prove it on one fully-developed project.

Backend:

- Extract `create_project(db, payload, actor)` into `app/services/projects.py`; the route delegates to it. Record a `create_project` contribution (Decision #5 / closes a small provenance gap — `primitives.md` lists "created a project" as a contribution). Behaviour-preserving for existing callers.
- Add `backend/scripts/seed_demo.py` (runnable as `uv run python -m scripts.seed_demo`): opens an `AsyncSession`, creates demo `Actor`s, and authors content purely through the services. Idempotent (skip-if-present, keyed on demo project title); `--reset` flag does a guarded, demo-scoped teardown (Decision #5).
- Author **demo project #1 — dark matter model constraints** to the full bar in Decision #6 (threads; mixed-kind claims; pinned evidence; ≥1 artifact + an artifact validation; ≥1 contested claim; ≥1 dead-end branch; a coherent checkpoint timeline).

Tests:

- `create_project` service records the project + a `create_project` contribution; the route still returns `201` with the same shape;
- a seed smoke test: running the harness against a clean DB produces the expected counts (threads/claims/evidence/artifacts/checkpoints/validations/branches), a contested claim, and a dead-end branch; a second run is a no-op (idempotency); `--reset` then reseed reproduces identical counts.

Exit criteria:

- `uv run python -m scripts.seed_demo` against a clean migrated DB yields one rich, internally-consistent demo project visible through the existing read APIs, and is safe to re-run.

### `0.5.3` — Second Demo + Integrity Coverage

Add variety to the project index and stress the harness across domains.

- Author **demo project #2 — high-Tc superconductivity mechanisms** to the same Decision #6 bar, reusing the harness (no new mechanism — content only).
- Ensure both demos collectively exercise every primitive and every `ValidationOutcome` that makes narrative sense (`passed`, `failed`, `inconclusive`, `contradicts`, `retract`), and that the artifact↔claim relation kinds (`expresses` / `supports` / `derived_from`) all appear.
- Extend the seed smoke test to cover both projects and assert the cross-project counts on the project index/overview reads.

Exit criteria:

- Two distinct, believable research projects exist; the project index reads as a research platform with real activity, and the integrity layer (contradictions, dead-ends, validated artifacts) is visible in both.

### `0.5.4` — Frontend Polish + Live Verification

Make the seeded richness render well and document the live walkthrough.

Frontend:

- Project index: present the two demos clearly (research question, lightweight activity summary from the overview counts). Minimal — reuse existing components.
- Render **artifacts** on claims in the workspace (kind + name + linked relation + a hash/URI affordance), reusing the evidence-rendering pattern; add an artifacts read to `lib/api.ts` and the matching types/query-keys.

Verification:

- Document a single end-to-end **live** verification path in `docs/changelog.md`: provision/confirm the DB → `alembic upgrade head` → `seed_demo` → open the deployed frontend → read both projects → confirm a contested claim, a dead-end branch, and a validated artifact are all visible.
- `npm run typecheck` / `lint` / `build` pass; backend `ruff` + `pytest` (now mostly *running*, not skipping) pass.

Exit criteria:

- A visitor with zero setup sees two living research maps; artifacts render on claims; the documented live path reproduces the whole demo from a clean database.

## Cross-Cutting Concerns

- **Single chokepoint preserved.** Artifact creation is recorded *through* `CheckpointService.create_checkpoint` (service-supplied refs + `contribution_action`), exactly like validation/branch in `0.4.x`. The chokepoint stays the sole producer of checkpoints and owns the single commit; the seed harness composes service calls and never mints checkpoints itself.
- **All seeded writes go through the service layer.** No Core bulk inserts for ledger content (`CLAUDE.md`); the only sanctioned Core access is the guarded `--reset` teardown (demo-scoped, dev-only).
- **Append-only.** Guard set unchanged (`Checkpoint`, `CheckpointRef`, `FundingAllocation`, `Validation`). `Artifact` stays mutable by design (updatable through a checkpoint per `primitives.md`); `0.5.0` only creates them.
- **Provenance is real.** Every seeded row is attributed to a demo `Actor` and carries its contribution — the demo *is* a provenance demo, so seeded history must be indistinguishable in kind from human-authored history.
- **Schema migrations.** Exactly one: `0004_claim_artifact_links` (`down_revision = 0003`). Verified offline and against the live DB.
- **Actor identity.** Unchanged — demo actors via the existing `X-Dev-Actor-Id` model; real identity arrives in `0.6.0`.
- **Live database from here on.** Unlike `0.3.x`/`0.4.x` (verified offline, tests skipped without a DB), `0.5.x` runs against the prerequisite's live Postgres — DB-backed tests are expected to execute and pass, not skip.

## Definition Of Done For `0.5.0`

`0.5.0` ships when, against the live migrated database, a single documented command seeds **two** realistic research projects — dark matter model constraints and high-Tc superconductivity — each carrying threads, mixed-kind claims, pinned evidence, at least one produced **artifact** with a validation, at least one **contested claim**, and at least one preserved **dead-end branch**; the seed is idempotent and runs entirely through the service layer with full provenance; the workspace and project index render all of it; and the `Artifact` write path (incl. artifact-validation) is covered by passing, actually-executed tests. At that point OpenTheory presents as a living research map to a first-time visitor with zero setup.
