# 0.3.0 — Implementation Plan (Phase Overview)

## Purpose

This document scopes the four implementation phases that together deliver `0.3.0 — Human-Operable Research Ledger`. It is the parent plan: each phase listed here will be expanded into its own dedicated, detailed implementation document before work begins.

The goal at this level is to:

- Lock in the boundary, dependencies, and exit criteria of each phase.
- Surface cross-cutting concerns that must be agreed before phase plans are written.
- Provide a single map of the release so phase plans can stay focused without re-litigating scope.

## Release Recap

`0.3.0` activates the primitives scaffolded in `0.1.0` (`Thread`, `Claim`, `Evidence`, `Checkpoint`, `Contribution`) so a human user can perform a research move end-to-end through the UI and see a durable, attributed checkpoint for it.

No new domain primitives are introduced. The work is: API write paths, a single append-only checkpoint service, automatic contribution recording, the first real Alembic migration, and a three-panel research workspace UI.

## Phase Map

| Phase   | Title                                | Stack focus      | Output                                                        |
| ------- | ------------------------------------ | ---------------- | ------------------------------------------------------------- |
| `0.3.1` | Backend Write Path                   | Backend + DB     | Threads/Claims/Evidence APIs, schema joins, first migration   |
| `0.3.2` | Checkpoint Service & Append-Only     | Backend          | Checkpoint service chokepoint, contributions, ledger writes   |
| `0.3.3` | Frontend Research Workspace          | Frontend         | Three-panel project workspace wired to all create/read flows  |
| `0.3.4` | Ledger Read Model & Polish           | Backend + Frontend | Enriched read models, ledger-shaped timeline, aggregate counts |

Each phase ends in a deployable, demoable state.

## Phase 0.3.1 — Backend Write Path: Threads, Claims, Evidence

### Intent

Stand up the create/read API surface for the three primitives that compose a research move, plus the two relational tables needed to make those relations queryable. No checkpoint behavior yet.

### Scope

- **Schema**
  - Add `claim_evidence_links` join table with a `relation_kind` `VARCHAR(20)` column (allowed values `support`, `weaken`, `context` validated in the service layer; not promoted to a Postgres enum yet).
  - Add `checkpoint_refs` join table with polymorphic `target_type` (`VARCHAR(20)`), `target_id` (UUID), and `role` (`VARCHAR(40)`) columns. Introduced in `0.3.1` even though it is consumed in `0.3.2`, to avoid a follow-up migration.
  - Add minimal `POST /api/v1/actors` and `GET /api/v1/actors` so dev actors can be created by hand during testing. No auto-seeding.
  - Generate a single baseline Alembic migration covering all existing `0.1.0` models plus the two new join tables. No data seeding inside the migration.
- **API**
  - Actor create + list (manual dev actor management).
  - Thread create + list under a project, thread detail.
  - Claim create + list under a thread, claim detail.
  - Evidence create scoped to a claim (creates `Evidence` row + `claim_evidence_links` row in one transaction), evidence list joined through the link table. Evidence rows are reusable across claims via additional `claim_evidence_links` rows in later requests.
  - All write endpoints accept the acting actor via `X-Dev-Actor-Id` header (resolved to an existing `Actor` row); requests without a valid header are rejected.
- **Service layer**
  - Thin create services for thread, claim, and evidence. No checkpoint interaction yet.
- **Tests**
  - Create flows for each primitive, including each `relation_kind` for evidence.
  - Read endpoints return the relation kind on evidence reads.

### Out of scope

- Any checkpoint creation, contribution recording, or append-only enforcement (deferred to `0.3.2`).
- Any frontend work (deferred to `0.3.3`).

### Exit criteria

- Migration applies cleanly against an empty database.
- All create/read endpoints return the documented payloads, including `relation_kind` on evidence reads.
- Backend test suite green; `ruff` clean.

### Dependencies

- None beyond the `0.1.0` model scaffold.

## Phase 0.3.2 — Checkpoint Service And Append-Only Enforcement

### Intent

Make the ledger real. Introduce the checkpoint primitive as the only sanctioned way to record a meaningful research state change, enforced at a single service-layer chokepoint.

### Scope

- **Service**
  - `CheckpointService.create_checkpoint` as the sole entry point that:
    - Validates project and optional thread context.
    - Accepts a free-form JSON `content` payload authored by the user (notes, summary, anything). No structured schema is enforced beyond "valid JSON".
    - Writes one `checkpoint_refs` row per referenced claim, evidence, artifact, or thread, with a `role` describing the action.
    - Links parent checkpoints via `checkpoint_parents`.
    - Rejects update/delete on checkpoints and `checkpoint_refs`.
  - Checkpoints are always created explicitly by a user action. Thread/claim/evidence create flows do **not** auto-promote to a checkpoint; promotion is a separate, deliberate user step. Automated checkpoint creation by a review agent is deferred to a later release.
- **API**
  - Checkpoint create under a project, checkpoint list under a project, checkpoint detail.
- **Contributions**
  - Auto-record a `Contribution` for checkpoint creation.
  - Back-fill auto-recording of `Contribution` for thread, claim, and evidence creates introduced in `0.3.1`, now that checkpoints are the canonical event.
- **Tests**
  - Checkpoint creation with parent linkage and references.
  - Rejection of update/delete attempts.
  - Contribution presence for all four create flows.

### Out of scope

- Validation records, branch lifecycle, artifact write flows (later releases).
- Frontend surfacing of checkpoints (deferred to `0.3.3`).

### Exit criteria

- A single service-layer entry point is the only producer of checkpoints.
- Append-only behavior is covered by tests.
- Contributions exist for all four create flows.

### Dependencies

- `0.3.1` schema (specifically `checkpoint_refs`) and write paths.

## Phase 0.3.3 — Frontend Research Workspace

### Intent

Surface the new write paths in the product. Expand `/projects/[projectId]` from a read-only detail page into a three-panel workspace where a user can complete the full research move without dropping to the API.

### Scope

- **Layout**
  - Left: thread list with create-thread action.
  - Center: selected thread workspace with claims, attached evidence, create-claim and attach-evidence actions.
  - Right: project checkpoint timeline (newest first) with create-checkpoint action scoped to the current thread context.
- **Typed API access**
  - Reads: project threads, thread detail, claims for a thread, evidence for a claim, checkpoints for a project.
  - Writes: the four create flows from `0.3.1` and `0.3.2`, using TanStack Query mutations with query-key invalidation for affected reads.
- **Dev actor selection**
  - A small dev-actor switcher in the workspace shell that lists actors from `GET /api/v1/actors`, lets the user pick or create one, persists the selection in `localStorage`, and attaches it as `X-Dev-Actor-Id` on all write requests. Replaced by real auth in `0.6.0`.
- **States**
  - Loading, empty, and error states for each panel.
- **UX constraints**
  - Minimal create UIs (single-line title + optional description). No rich editors.

### Out of scope

- Enriched ledger read model (deferred to `0.3.4`).
- DAG visualization, validation UI, branch UI.

### Exit criteria

- A user can complete the full success-criterion flow end-to-end through the UI.
- Each panel handles loading, empty, and error states.
- `npm run typecheck`, `npm run lint`, and `npm run build` all pass.

### Dependencies

- `0.3.1` and `0.3.2` API surfaces.

## Phase 0.3.4 — Ledger Read Model And Polish

### Intent

Make the checkpoint timeline read like a real research record, not a generic activity feed. Tighten the loop between the actions a user takes and what the project page shows back.

### Scope

- **Backend read models**
  - Project detail enriched with threads (with claim counts), most recent checkpoints (bounded count), and aggregate counts (threads, claims, evidence, checkpoints).
  - Checkpoint read model enriched with creating actor, contribution kind, and referenced claim/evidence summaries (id, title, relation kind).
- **Frontend rendering**
  - Checkpoint timeline shows actor, action summary, and referenced claims/evidence.
  - Workspace header shows project-level aggregate counts.
  - Empty-state copy explains the next research move a user can make.
- **Verification**
  - Documented end-to-end manual verification path appended to the changelog.

### Out of scope

- Any new write paths.
- Real authentication or attribution beyond the dev actor.

### Exit criteria

- The project page communicates the state of the ledger at a glance.
- A new user can understand what a checkpoint records without reading the docs.
- All previously listed tooling commands still pass on both backend and frontend.

### Dependencies

- `0.3.3` workspace shell as the host surface.

## Cross-Cutting Concerns

These apply across all four phases and should be settled in this parent plan rather than re-decided in each phase doc.

- **Actor identity.** No authentication in `0.3.X`. Actors are created manually via the dev API and selected per request via the `X-Dev-Actor-Id` header. No global "current user" assumption, no startup seeding. Real identity arrives in `0.6.0`.
- **Manual-first, automation later.** All meaningful state changes in `0.3.X` are explicit user actions. Automated checkpointing by a review agent, autonomous thread spawning, and similar agent behaviors are out of scope and tracked separately.
- **Enums deferred.** New string-typed columns (`relation_kind`, `target_type`, `role`) are stored as `VARCHAR` with allowed values validated in the service layer. We will revisit promotion to Postgres enums once the vocabulary stabilizes; the early phase of the project favors flexibility over schema rigidity.
- **Append-only invariants.** Enforced exclusively in `CheckpointService`. No route handler or migration mutates existing checkpoints or `checkpoint_refs`.
- **Migrations.** `0.3.1` produces a single baseline migration covering all `0.1.0` models plus the two new join tables. Later phases add migrations only if column-level changes are introduced (e.g., contribution metadata columns).
- **Frontend state.** TanStack Query covers both reads and write mutations (with query-key invalidation). No additional client state library is introduced in `0.3.X`.
- **Testing.** Each phase ships its tests in the same commits that introduce behavior. No phase exits with a red suite.
- **Telemetry / logging.** Out of scope. A single structured log line per checkpoint creation is sufficient for `0.3.X`.
- **Versioning.** Each sub-phase ships under its own `0.3.x` changelog entry, even when the user-visible change is small (e.g., `0.3.2` is backend-only).

## Definition Of Done For `0.3.0`

`0.3.0` ships when all four phases are merged and the success criterion in `0.3.0-human-operable-research-ledger.md` is reproducible against a clean database from the UI alone.

## Resolved Decisions

These were open questions during scoping; they are settled here so phase plans can proceed without re-litigation.

1. **Dev actor identity.** No auto-seeding. Actors are created manually via `POST /api/v1/actors`. Write requests must carry `X-Dev-Actor-Id`; the backend resolves it to an existing `Actor` row and rejects unknown IDs. The frontend dev-actor switcher persists the selection in `localStorage`.
2. **Checkpoint content payload.** Free-form JSON authored by the user. No structured derivation in `0.3.X`.
3. **Checkpoint creation is manual.** Thread/claim/evidence creates do not auto-promote to checkpoints. A future review agent will own automated checkpointing; that work is scoped separately and is not part of `0.3.X`.
4. **Enums deferred.** `relation_kind`, `target_type`, and `role` are stored as `VARCHAR` columns with service-layer validation. Promotion to Postgres enums is revisited once vocabulary stabilizes.
5. **Evidence is reusable across claims.** `claim_evidence_links` is a true many-to-many. The same evidence row can be attached to multiple claims with potentially different relation kinds.
6. **`0.3.4` stays as its own phase.** Splitting the read-model enrichment from the initial workspace wiring keeps each phase narrow and reduces the risk of UI rework when the enriched payloads land. Net effect on bug surface is favorable.
7. **TanStack Query for reads and writes.** Confirmed compatible with the Next.js App Router. Mutations + query-key invalidation cover the four create flows; no additional client state library is introduced in `0.3.X`.
8. **Single baseline migration in `0.3.1`.** All `0.1.0` models plus `claim_evidence_links` and `checkpoint_refs` ship in one Alembic revision. There is no production data to protect, so a split adds maintenance overhead without payoff.
