# 0.6.0 — Authentication And Funding

## Goal

Give OpenTheory **real identity** and a **real funding write path**, in that order.

Today every write is attributed by an unverified `X-Dev-Actor-Id` header — any caller can act as
any actor (`api/deps.py`). That is a deliberate `0.3.x` stopgap, and it is the single blocker
before money or compute can be attributed to anyone. This release replaces the dev header with
verified identity mapped onto the existing `Actor` model, then activates the **`FundingAllocation`**
primitive — the last core primitive with a model but no write path (after `Artifact`).

Funding is built **money-stubbed and source-aware**: a `FundingAllocation` records a budget
committed to a project, tagged with its `source` (`native` = Kamino comps the budget against the
platform's own compute; `stripe` = an external funder pays). **Native funding is made real**
(Kamino can fund its own projects now); **real Stripe settlement and real compute *spend* are
deferred** — spend has no meaning until agents execute (`0.7.0`), and there is no point taking
external money for a budget that nothing draws against yet.

This release does **not** introduce: real Stripe charges/webhooks, per-funder API keys, compute
metering/budget *debiting*, reputation/influence scoring, branch merge, or autonomous agents. The
shared `OPENROUTER_API_KEY` is platform infrastructure (a backend env var, like `DATABASE_URL`) and
arrives with agents in `0.7.0`, **not here** — it would be inert.

## Prerequisite (tracked separately, the tail of `0.4.0` — not a sub-phase of `0.6.0`)

`0.6.0` assumes the **`0.4.0` Definition-of-Done gate is closed**: the 31 DB-backed tests run green
against a *separate* test Postgres (`TEST_DATABASE_URL`, never the live/seeded DB — the fixture
`drop_all`s). Through `0.4.7` the entire service layer — the checkpoint chokepoint, the append-only
guards, contribution recording — has only ever been verified *offline* (`pytest` → `5 passed, 31
skipped`). Auth and funding compose with that service layer; building on it unverified is building on
sand. **Confirm this gate is closed before `0.6.1` ships.** It is a release/infra task (carried as
the tail of `0.4.0`), not a deliverable of this plan.

## Success Criterion

A visitor to the deployed product can:

1. **Sign in** with a real identity (OAuth / email) and see themselves as the acting actor — no dev
   header, no actor picker. Their identity maps to exactly one `Actor` row.
2. Perform every existing write (thread, claim, evidence, checkpoint, validation, branch) **as their
   authenticated self**; attribution and `Contribution` provenance are unchanged in shape but now
   carry a *verified* actor.
3. As a **Kamino (internal) member**, **fund** one of their projects via **native funding** — a
   `FundingAllocation` (`source=native`, `status=settled`) is recorded, attributed by a `fund`
   `Contribution`, and the project's **budget** + **funding history** update.
4. See the **funder / contributor / validator separation** hold: funding a project grants budget and
   nothing else — no contribution credit toward findings, no validation authority.
5. Be **rejected** when attempting to act unauthenticated (`401`), or when a non-internal actor
   attempts native funding (`403`).

Append-only and provenance guarantees are identical to a human research action: the funding ledger is
immutable, every funding event is attributed, and the *research* checkpoint DAG is untouched by
financial events.

## Scope Summary

In scope:

- **Verified identity**: replace the `X-Dev-Actor-Id` dependency internals with bearer-JWT
  verification mapped onto `Actor.external_id`; just-in-time (JIT) actor provisioning on first login;
  the `ActingActor` *contract* (a resolved `Actor`) is unchanged, so the service layer is untouched.
- **Roles**: a minimal `actors.roles` column for queryable authorization; an `internal` (Kamino) role
  gates native funding. One migration.
- **Frontend auth**: sign-in/out, session, a bearer token attached to every request, write
  affordances gated on auth, the dev-actor switcher retired (kept behind a dev flag).
- **`FundingAllocation` write path**: a `funding` service that records an allocation + a `fund`
  `Contribution` (contribution-only — **not** through the checkpoint chokepoint; Decision #3),
  source-aware (`native` / `stripe`), API routes, read models. One migration (`source` column).
- **Budget read models**: project overview gains funded totals (by source/status) and a funding
  history list; spend is `0` until `0.7.0`.
- **Frontend funding surfaces**: a native-funding (top-up) flow for internal actors, project budget +
  funding-history views, the role separation made legible.

Out of scope (deferred):

- **Real Stripe charges, Checkout, and webhook settlement** — the `stripe` source is *modeled* and
  creatable as `pending`, but no money moves and no webhook lands here. Real settlement is a dedicated
  later release (Decision #5).
- **Compute metering / budget *debiting*** — `FundingAllocation` is *authorization* (budget granted);
  the *spend* ledger that debits it as agents consume tokens is `0.7.0` (Decision #6). `OPENROUTER_API_KEY`
  (shared platform infra) arrives then.
- **Reputation / influence scoring**, **branch merge**, **autonomous agents**, **per-funder API keys**,
  **object storage** — unchanged deferrals from `0.4.0`/`0.5.0`.
- **Artifact write path** — still open from the descoped `0.5.0`; not on this release's critical path
  (called out in Cross-Cutting Concerns so it isn't lost).

## Primitives Activated

| Primitive          | Already modeled | New in `0.6.X`                                                                              |
| ------------------ | --------------- | -------------------------------------------------------------------------------------------- |
| `Actor`            | yes (`0.1.0`)   | `external_id` used as the verified-identity key; JIT provisioning; new `roles` column         |
| `FundingAllocation`| yes (`0.1.0`)   | the last write path: source-aware create + read, budget read models, a `fund` contribution    |
| `Contribution`     | yes             | auto-recorded for `fund` (via the existing `funding_allocation_id` FK; `checkpoint_id` NULL)   |

## Resolved Decisions

### Decision #1 — Real identity maps onto `Actor.external_id`; the `ActingActor` contract is preserved

The `Actor` model already carries `external_id` (`String(255)`, **unique**, nullable) — the
purpose-built hook for an external identity-provider subject. Auth therefore does **not** introduce a
parallel user table.

**Recommendation:** one `Actor` per authenticated human, `Actor.external_id = <IdP subject>`,
`type = human`. The auth dependency (1) extracts and **verifies** the bearer JWT (signature, issuer,
audience, expiry), (2) reads `sub` + `email`, (3) looks up the `Actor` by `external_id == sub`, and
(4) **JIT-provisions** one if absent (idempotent on the unique `external_id`). It returns an `Actor`
— *exactly* what `get_acting_actor` returns today. The `ActingActor` type alias and every service
signature are unchanged; only the *internals* of `get_acting_actor` change. The entire service layer
(`services/*.py`) is untouched.

*Rejected alternative:* a separate `User` table joined to `Actor`. Rejected — `external_id` was added
for this; a second identity table duplicates state and contradicts "humans and agents use the same
actor model" (`primitives.md`). An `agent` actor (`0.7.0`) gets credentials a different way; it does
not log in through the human IdP.

### Decision #2 — Supabase Auth as the IdP, behind a swappable verification adapter

The stack already runs on **Supabase Postgres** (`techstack.md`), and Supabase provides managed auth
(OAuth providers + email/password, JWT issuance) at no added vendor. The backend stays decoupled: it
**only verifies a JWT and reads claims** — it does not couple domain logic to the Supabase client.

**Recommendation:** Supabase Auth issues the session JWT; the backend verifies it (project JWT
secret / JWKS) in a single small adapter (`core/auth.py`) that turns a bearer token into
`(subject, email)`. Swapping to Clerk/Auth0 later changes only that adapter (all three are
JWT/JWKS-based) — the dependency, the `Actor` mapping, and the services are identical.

*This is the one genuinely swappable choice in the release.* If you prefer Clerk/Auth0, only
`core/auth.py` + config change; the rest of the plan stands.

### Decision #3 — Funding is contribution-only; it does **not** mint a research checkpoint

`primitives.md` states `Validation` "**is recorded through `Checkpoint`**" — which is why validation
and branch flows compose with `CheckpointService.create_checkpoint` (`0.4.x`). `FundingAllocation`
carries **no such language**: it is "a **financial** ledger entry," and `Contribution` references
"a `Checkpoint`" **or** "a `FundingAllocation`" as alternatives. `Contribution` already has a
dedicated `funding_allocation_id` FK (`contribution.py:29`) for exactly this.

**Recommendation:** a funding event writes a `FundingAllocation` row and one `fund` `Contribution`
with `funding_allocation_id` set and `checkpoint_id = NULL` — the **contribution-only** pattern used
by thread/claim/evidence creates, **not** the checkpoint-recorded pattern. The funding *service* owns
the single `commit` (calling `record_contribution(...)`, which does not commit). No checkpoint is
minted; the **research DAG stays research-only**.

*Rationale:* funding is deliberately separate from the research ledger (`primitives.md`: funding
"does not imply correctness, authorship, or validation"). Minting a research checkpoint for "Alice
funded $500" would pollute the checkpoint timeline with financial events and conflate the financial
ledger with the research ledger. Append-only is already guaranteed — `FundingAllocation` is in the
ORM guard set (`models/append_only.py`) — so funding needs no checkpoint to be immutable.

*Rejected alternative:* record funding through the chokepoint like validation. Rejected: it conflates
the two ledgers and contradicts the funding/contribution/validation separation that is the point of
the data model.

*Implementation note:* extend `contributions.record_contribution` to accept an optional
`funding_allocation_id` (pass-through; default `None`). Add `ACTION_FUND = "fund"`. No change to the
chokepoint.

### Decision #4 — A `source` discriminator (`native` / `stripe`); native funding is gated to internal actors

`FundingKind` (`top_up` / `grant` / `refund` / `adjustment`) is the *accounting category*, not where
the money came from. "Kamino comps it" vs "an external funder pays" is an orthogonal axis with no
column today.

**Recommendation:** add a `FundingSource` enum (`native` | `stripe`) and a `funding_allocations.source`
column (named PG enum `funding_source`, mirroring `funding_kind`). **`native`** = Kamino grants budget
backed by the platform's own (future) compute; creatable **only** by an actor holding the `internal`
(Kamino) role (`403` otherwise). **`stripe`** = external paid funding; modeled and creatable as
`pending`, but no real settlement here (Decision #5).

The `internal` role lives in the new `actors.roles` column (Decision #1's migration companion). On
JIT provisioning, an actor whose verified email is in a config allowlist (`internal_actor_emails`, or
an internal email domain) is granted `internal` — the pragmatic bootstrap for "Kamino members like
me." Roles are otherwise assignable out-of-band.

*Rejected alternative:* store the role in `actor_metadata` (JSON, no migration). Rejected — authz
should be a queryable, indexable column, and `roles` will be reused for validator/agent permissions
later. One small column is worth it.

### Decision #5 — `FundingAllocation` stays append-only; allocations are created in their terminal status

`FundingAllocation` is append-only (`primitives.md`; ORM guard set), so a `pending → settled`
transition **cannot be an in-place update**. This is a feature, not a problem — the system handles
every correction as a *new row*, never an edit.

**Recommendation:** an allocation is **born in its terminal status**. `native` allocations are created
`settled` (no money moves — Kamino is comping budget). Real `stripe` settlement is deferred to a later
release and, when it lands, will **create the `settled` row on webhook confirmation** (create-on-confirm)
rather than mutating a `pending` one. `refund`/`adjustment` are likewise future *new* allocations
(negative or offsetting, `kind=refund`, referencing the original via `payment_reference`/`notes`),
never edits. `pending`/`failed`/`refunded` stay in `FundingStatus` for those flows. `FundingAllocation`
is **not** removed from the append-only guard set.

*Consequence for this release:* the only allocations actually *created* are `native`/`settled` (the
real, demoable path) and optionally `stripe`/`pending` (modeled, inert). **Budget = sum of `settled`
allocations.**

### Decision #6 — Funding is *authorization*, not *spend*; metering is `0.7.0`

A `FundingAllocation` records budget *granted*. **Nothing debits it** in `0.6.0` — there are no agents
to consume compute. The OpenRouter key is shared platform infrastructure (one backend env var; every
agent call, whoever funded the work, runs against it), and it is inert until `0.7.0`.

**Recommendation:** model budget as `funded = Σ settled allocations`, `spent = 0`,
`available = funded`. Do **not** add a spend/usage ledger or `OPENROUTER_API_KEY` config here. When
agents land (`0.7.0`), a **separate** append-only usage/debit ledger will record token consumption and
compute `available = funded − spent`; because `FundingAllocation` is append-only, spend is **never**
modeled by mutating `amount` — it is its own ledger. Flagged now so no one is tempted to make `amount`
mutable later.

## Sub-Phases

Four sub-phases. Auth ships as a complete vertical slice (`0.6.1` backend + `0.6.2` frontend) **before**
funding begins (funding depends on verified identity). Each phase ends deployable and adds its tests in
the same commit. Unlike `0.3.x`/`0.4.x`, these run against the **live, migrated database** from the
prerequisite — DB-backed tests are expected to execute, not skip.

---

### `0.6.1` — Verified Identity (Backend)

Replace the dev-header attribution with verified-JWT identity, preserving the `ActingActor` contract.

Schema (migration `0004_actor_roles`, `down_revision = 0003_checkpoint_branch_id`):

- Add `actors.roles` (`ARRAY(String)`, server default empty, not null). Backfill is a no-op (existing
  actors get `[]`). Export remains via `models/__init__.py` (`Actor` already exported).

Backend:

- Add `app/core/auth.py` — the swappable verification adapter: verify a bearer JWT (signature via the
  Supabase JWT secret/JWKS, plus issuer/audience/expiry) and return `(subject, email)`. Verification
  failure → `401`.
- Rewrite `get_acting_actor` (`api/deps.py`) internals: read the `Authorization: Bearer` token →
  verify via `core/auth.py` → look up `Actor` by `external_id == subject` → **JIT-provision** if absent
  (`type=human`, `display_name` from token name/email, `external_id=subject`, email into
  `actor_metadata`, `roles=["internal"]` iff the email matches `internal_actor_emails`). Return the
  `Actor`. **The return type and every downstream service are unchanged.**
- Keep a **dev-header fallback** behind a config flag (`auth_dev_header_enabled`, default `False`;
  `True` in local/test) so local development and the DB-backed suite keep working without an IdP. In
  production the flag is off and only a verified token is accepted.
- Add config (`core/config.py`): `supabase_jwt_secret` / `supabase_jwks_url`, `supabase_project_url`,
  `auth_dev_header_enabled`, `internal_actor_emails` (comma-split, like `backend_cors_origins`).
- Retire the open `POST /actors` bootstrap route (anyone can currently mint actors) — provisioning is
  JIT now; keep it behind `auth_dev_header_enabled` for tests. Add `GET /me` (the resolved actor +
  roles) for the frontend.
- Add an `internal`-role helper (`require_internal`) for later authz reuse (used in `0.6.3`).

Tests (against the live DB):

- a valid token provisions exactly one `Actor` (JIT) and resolves it; a second request with the same
  `sub` reuses it (no duplicate; unique `external_id`).
- a malformed/expired/wrong-audience token → `401`; a missing token → `401`.
- with `auth_dev_header_enabled=True`, the `X-Dev-Actor-Id` path still resolves (test/local parity).
- an email in `internal_actor_emails` provisions with `roles=["internal"]`; others with `[]`.
- existing write flows (thread/claim/validation/...) attribute to the JIT actor unchanged — a
  contribution is recorded with the resolved `actor_id`.
- migration `0004` loads as head and round-trips (offline + live).

Exit criteria:

- A verified token resolves (or provisions) exactly one `Actor`; unauthenticated writes are rejected;
  the service layer is byte-for-byte unchanged in behavior; the dev-header path survives only behind
  the dev flag.

### `0.6.2` — Frontend Auth

Make sign-in real and attach the verified token to every request.

Frontend:

- Add Supabase Auth (`@supabase/supabase-js` + `@supabase/ssr`): sign-in/out (OAuth + email), session
  persisted in cookies, an `AuthProvider` exposing the current user; replace `providers/dev-actor-provider`.
- Attach `Authorization: Bearer <access_token>` on **every** call in `lib/api.ts`'s `request` helper
  (not just writes). Simplify `writeInit(actorId, body)` → `writeInit(body)` — the acting actor now
  derives from the token server-side; drop the `actorId` argument threaded through every mutation.
- Replace `DevActorSwitcher` in `site-header.tsx` with a signed-in identity menu (name + sign-out),
  driven by `GET /me`. Keep `DevActorSwitcher` mounted only when a public dev flag is set
  (`NEXT_PUBLIC_AUTH_DEV`), for local work.
- Gate write affordances on auth (signed-out users read; writing prompts sign-in). Loading/empty/error
  states via the existing `panel-state` helper.

Tests / verification:

- `npm run typecheck`, `lint`, `build` pass after removing the `actorId` parameter everywhere.
- Manual: sign in → create a thread/claim → it is attributed to the signed-in actor; sign out → write
  affordances are gated; refresh preserves the session.

Exit criteria:

- A user signs in and performs the full `0.3.0`–`0.4.0` workflow as their verified self, with no dev
  header and no actor picker; the token rides every request.

### `0.6.3` — Funding Write Path (Backend, money-stubbed, source-aware)

Activate the last unwritten core primitive — `FundingAllocation` — as a source-aware, contribution-only
ledger write.

Schema (migration `0005_funding_source`, `down_revision = 0004_actor_roles`):

- Add `FundingSource` enum (`native` | `stripe`) to `models/enums.py` (named PG type `funding_source`).
- Add `funding_allocations.source` (`Enum(FundingSource, name="funding_source")`, not null). Existing
  rows: none yet, so no backfill concern; if any exist, default them `native` in the migration.

Backend:

- Add `app/services/funding.py` with `create_funding(db, project_id, payload, actor)` that, in one
  transaction (Decision #3): validates the project; validates `source` and, for `source=native`,
  **requires the actor to hold the `internal` role** (`403` otherwise); writes the `FundingAllocation`
  (`actor_id`, `amount`, `currency`, `kind`, `source`, `status` — `settled` for native; `pending` for
  stripe per Decision #5); records a `fund` `Contribution` via `record_contribution(...,
  funding_allocation_id=allocation.id, checkpoint_id=None)`; **the service owns the single commit**.
  Plus read helpers: `list_funding(project_id)`, `get_funding(id)`, and `project_budget(project_id)`
  (`funded` = Σ settled, `spent` = 0, `available` = funded, with by-source/by-status breakdowns).
- Add `app/schemas/funding.py` (`FundingCreate`, `FundingRead`, `ProjectBudget`). Read includes
  amount/currency/kind/source/status, actor summary, created_at.
- Add `ACTION_FUND = "fund"` and the `funding_allocation_id` pass-through to `contributions.py`.
- Add API routes:
  - `POST /api/v1/projects/{project_id}/funding` (requires auth; native requires `internal`)
  - `GET  /api/v1/projects/{project_id}/funding` (funding history)
  - `GET  /api/v1/projects/{project_id}/budget` (budget summary)
  - `GET  /api/v1/funding/{funding_id}`
- Extend `get_project_overview` (`services/projects.py`) with a `budget` block (funded/spent/available)
  — reusing the `_count`/grouped-query pattern; no N+1.
- **Do not** add `funding_allocation` to the checkpoint-ref vocabulary (`_REF_TARGET_MODELS`) — funding
  does not produce checkpoint refs (Decision #3).

Tests (against the live DB):

- native funding by an `internal` actor records one `settled` `FundingAllocation` + one `fund`
  `Contribution` (with `funding_allocation_id` set, `checkpoint_id` NULL), and **mints no checkpoint**
  (the checkpoint count is unchanged — locks Decision #3).
- native funding by a non-internal actor → `403`; unauthenticated → `401`.
- stripe-source funding is created `pending`; budget counts only `settled` (so it does not inflate
  `funded`).
- an ORM update/delete of a `FundingAllocation` raises `AppendOnlyError`; no mutation route exists.
- `project_budget` returns `funded = Σ settled`, `spent = 0`, `available = funded`, with correct
  by-source/by-status splits.
- the funder/contributor/validator separation: funding records a `fund` contribution but **no**
  authorship/validation contribution, and does not alter any claim/validation state.
- migration `0005` loads as head and round-trips (offline + live).

Exit criteria:

- An internal actor funds a project; the allocation is recorded, attributed, append-only, and
  reflected in the budget — with the research DAG and the validation/contribution separation untouched.

## Cross-Cutting Concerns

- **The service-layer contract is preserved.** Auth changes only `get_acting_actor`'s internals; every
  service still receives a resolved `Actor`. No service signature changes for auth.
- **Single chokepoint preserved — and funding stays out of it.** Funding is contribution-only
  (Decision #3); the checkpoint chokepoint remains the sole producer of checkpoints and is unchanged.
- **Append-only.** Guard set unchanged (`Checkpoint`, `CheckpointRef`, `FundingAllocation`,
  `Validation`). `FundingAllocation` stays guarded; status lifecycle is modeled as new rows, never
  edits (Decision #5). `Actor.roles` is mutable (identity attribute, like `Branch.status`), not a
  ledger event.
- **Funding ≠ contribution ≠ validation.** Enforced in the data model and surfaced in the UI: a
  `fund` contribution attributes budget only; it confers no authorship or validation (`primitives.md`).
- **Authorization is queryable.** `actors.roles` backs the `internal` gate and is the substrate for
  validator/agent permissions later. The `internal` grant bootstraps from an email allowlist.
- **Funding is authorization, not spend.** No metering, no debit ledger, no `OPENROUTER_API_KEY` here;
  all arrive with agents (`0.7.0`, Decision #6).
- **Schema migrations.** Exactly two: `0004_actor_roles` (`0.6.1`) and `0005_funding_source` (`0.6.3`),
  chained off `0003`. Verified offline and against the live DB.
- **Secrets.** The Supabase JWT secret / keys and `internal_actor_emails` live only in the gitignored
  `.env` (documented in `.env.example`), per the `0.4.6` connection-secrets posture.
- **Still open, carried forward.** The `Artifact` write path (the other modeled-but-unwritten
  primitive, descoped with `0.5.0`) remains open; not required for `0.6.0` but noted so it is a
  conscious deferral.

## Definition Of Done For `0.6.0`

`0.6.0` ships when, against the live migrated database: a user signs in with a real verified identity
(no dev header) and performs the full existing research workflow attributed to their `Actor`;
unauthenticated writes are rejected; a Kamino (`internal`) member native-funds a project and sees a
`settled`, attributed, append-only `FundingAllocation` reflected in the project's budget and funding
history; a non-internal actor is `403` on native funding; the funding event records a `fund`
contribution and **mints no research checkpoint** (the funding/contribution/validation separation holds
end-to-end); and the `Actor`/auth and `FundingAllocation` write paths are covered by passing,
actually-executed tests. Real Stripe settlement and compute spend remain deferred to a later release /
`0.7.0`.
